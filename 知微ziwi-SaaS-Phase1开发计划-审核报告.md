## 三、关键问题清单

### 🔴 问题 1（缺失）：审批引擎/审批流 — 缺少独立任务

| 维度 | 内容 |
|------|------|
| **依据** | 产品方案 V1.4 §2.1 平台基础服务明确包含 **"审批引擎"**；§9.1 明确 **"Phase 1 审批使用知微内置审批流"**；§10.2 明确 企业微信/钉钉"Phase 1 审批使用知微内置审批流，不依赖OA即可运行" |
| **现状** | 开发计划 M0 七个任务中**无任何任务涉及审批引擎或审批流**。消息中心（任务4）仅负责推送通知，未覆盖审批流程。 |
| **影响** | 安灯超时升级（任务21）、工价调整审批（Phase 2）、以及其他需要审批流程的场景在 Phase 1 将缺失内置审批能力。 |
| **建议** | 在 M0 基础平台中新增一个独立任务 **"审批引擎/内置审批流"**，验收条件建议：<br>- 审批模板定义（如安灯升级审批、报离职审批等）<br>- 审批流配置（审批节点、审批人/角色、会签/或签）<br>- 审批人通知（通过任务4消息中心）<br>- 审批历史追溯<br>- 无需对接外部OA（Phase 2 再做OA双向对接） |

### 🟡 问题 2（偏差）：离线预警通道与产品方案不一致

| 维度 | 内容 |
|------|------|
| **依据** | 产品方案 V1.4 §13.3.2：<br>- 第 1 天：**邮件 + 站内信**（无短信）<br>- 第 7 天：邮件 + 短信 + 站内信<br>- 第 21 天：邮件 + 短信 + 站内信<br>- 第 30 天：**邮件 + 短信 + 客服外呼** |
| **现状** | 开发计划任务6验收条件："第 1/7/21/30 天触发**邮件+短信+站内信**"<br>1) 第 1 天多了"短信"（产品方案无短信）<br>2) 第 30 天缺少"客服外呼"（产品方案要求外呼） |
| **影响** | 低。预警通道差异不影响核心逻辑，但与产品方案不一致可能导致客户体验偏离设计意图。 |
| **建议** | 按产品方案 V1.4 §13.3.2 修正：第1天仅邮件+站内信，第30天加入客服外呼触发逻辑。 |

### 🟡 问题 3（不明确）：API 网关熔断（ADR-003）实现无独立任务

| 维度 | 内容 |
|------|------|
| **依据** | 架构设计书 V1.4 §3.4-3.5 提供了完整的 **TenantCircuitBreaker + 紧急白名单 + 网关中间件** 实现设计（约120行代码级设计） |
| **现状** | 开发计划仅 ADR 清单中列出 ADR-003，任务1（脚手架）中引用为参考。**无独立实现任务**。 |
| **影响** | 熔断机制是 P0 架构决策（ADR-003），若仅在脚手架中作为"参考"可能导致被遗漏或实现不完整。 |
| **建议** | 方案A：在 M0 中新增独立任务"API网关熔断器实现"<br>方案B：在任务1（脚手架）或任务7（PgBouncer）验收条件中**显式写明熔断器实现要求**。<br>推荐方案A，因熔断逻辑（滑动窗口P95+状态机+白名单）复杂度足够独立为一个任务。 |

### 🟡 问题 4（偏差）：个人报工 UI 包含工价计算与预估

| 维度 | 内容 |
|------|------|
| **依据** | 产品方案 V1.4 §5.2.1：起步版（Phase 1 唯一可售版本）明确标注 **"工价配置：无（仅记录数量）"**。工价配置（计件/计时工资）属于 Phase 2 成长版范围。 |
| **现状** | 开发计划任务14验收条件中包含"工价：¥5.00/件"、"本单预估：合格×¥5.00=¥[ ]元"等工价计算逻辑。 |
| **影响** | 若 Phase 1 实现工价计算逻辑，将引入工价配置管理、版本管理、审批流程等 Phase 2 才需要的能力，导致 Phase 1 范围膨胀。 |
| **建议** | 任务14报工界面 Phase 1 仅记录 **合格数量 + 不良数量 + 工时**，不显示"工价"和"预估工资"字段。这些字段可**预埋 UI 占位但默认隐藏**，Phase 2 激活时再显示。 |

### 🟢 问题 5（建议优化）：ADR-024 小组分配 vs 产品方案边界

| 维度 | 内容 |
|------|------|
| **依据** | 架构评估 V1.4 §2（ADR-024）："小组分配采用暂估+补差两阶段模式。**Phase 1 仅支持系数均分**"。但产品方案 V1.4 Phase 1 范围明确**小组报工在 Phase 2**。 |
| **现状** | 开发计划正确跟随产品方案（Phase 1 只做个人报工）。但 ADR-024 的 Phase 1 声称与产品方案冲突。 |
| **影响** | 架构文档与产品方案不一致可能导致后续 Phase 2 开发时混淆。 |
| **建议** | 确认该 ADR 的 Phase 1 范围是否应修正为"Phase 1 不做，Phase 2 按暂估+补差模式实现"。 |

### 🟢 问题 6（建议优化）：任务时间线密集度

| 维度 | 内容 |
|------|------|
| **分析** | W5-W12 是最高负载期：M1（W3-W5）尾端 + M2（W5-W9）全期间 + M3（W9-W12）全期间，3 条业务模块流水线重叠，且 M2/M3 各含 5-6 个任务。 |
| **风险** | 前端（Vue3触屏/移动WAP）+ 后端（FastAPI）+ IoT（边缘网关）三线并行，9-12 人团队在 W5-W12 期间压力最大。 |
| **建议** | 1. 确认 W5-W12 期间前端工程师是否足够（UI/UX+触屏+WAP 三端并行设计）<br>2. 考虑将 M3 品质部分任务前置到 W7 启动（与安灯解耦），降低 W9-W12 峰值 |
