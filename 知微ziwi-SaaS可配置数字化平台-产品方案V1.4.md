## 一、产品定位与愿景

### 1.1 一句话定位

**知微 ziwi** —— 制造业轻量化数字化平台，客户在 ziwi.cn 自选模块组合，SaaS 即开即用，也可本地化部署。

### 1.2 核心价值主张

| 维度 | 价值 |
|------|------|
| **对中小企业** | 不需要百万预算、不需要半年实施，3~5 个模块快速上线，**1~2 个月上线（取决于您的数据准备情况）**，销售时按三档精准告知客户所属档位 |
| **对中大型企业** | 从核心模块起步，逐步扩展到全链路，已投的 ERP/CRM 不浪费，接口对接即可 |
| **对所有人** | 数据从车间到老板全链路贯通，告别信息孤岛 |

### 1.3 差异化竞争力

| 竞争维度 | 传统 MES/ERP | 知微 ziwi |
|---------|-------------|-----------|
| 采购方式 | 标书→招投标→签合同（3~6 月） | ziwi.cn 在线选配→签约→开通（1~2 周） |
| 模块选择 | 全家桶或定制开发 | 13 模块自由组合，推荐套餐打底 |
| 部署方式 | 纯本地部署 | SaaS 在线用 / 本地部署 / 两者兼有 |
| 实施周期 | 6~18 个月 | SaaS 1~4 周 / 本地 4~12 周 |
| 能碳融合 | 需额外采购或对接 | 原生内置能碳管理模块 |
| 终端覆盖 | PC 为主 | PC + 触屏终端 + PDA + 移动 WAP + 大屏 |

### 1.4 目标客户分层

| 层级 | 规模 | 典型场景 | 配置偏好 |
|------|------|---------|---------|
| **L1 入门版** | 50~150 人 | 无系统或仅有 Excel 管控 | 选择 2~3 个模块快速上线 |
| **L2 成长版** | 150~500 人 | 有部分系统但信息孤岛 | 选择 4~6 个模块打通链路 |
| **L3 旗舰版** | 500 人以上 | 有 ERP 但 MES/品质/仓储弱 | 全模块 + 第三方集成 + 本地部署 |

## 三、各模块功能边界定义

### M01 生产管理

**用户故事**：作为生产主管，我希望从接单到入库全流程在线管控，以便实时掌握生产进度和异常。

#### M01-1 制造模式与工厂类型

| 维度 | 离散型制造 | 流程型制造 | 对系统的影响 |
|------|----------|----------|------------|
| 典型行业 | 机加工、电子装配、汽车零部件 | 化工、食品饮料、制药、新材料 | 不同行业的报工模板需区分 |
| 生产单位 | 个、件、套（可数） | 公斤、升、吨、批次（连续） | 报工计量单位需支持"数量+单位"灵活配置 |
| 报工粒度 | 工序级（车→铣→磨→装配） | 批次级（投料→反应→出料） | 离散型需要工序级报工，流程型需要批次级报工 |
| 工时计量 | 按"工时"（小时/分钟） | 按"时间"或"产出速率" | 流程型更关注"运行时长"而非"人工工时" |
| 损耗/收率 | 不良品数量 | 收率百分比（产出/投入） | 流程型需要"收率"指标，离散型需要"不良率" |
| 设备绑定 | 一人多机 / 一机多人 | 连续生产线（设备链） | 流程型报工需关联整条产线而非单设备 |
| 倒班模式 | 按班次独立报工 | 连续生产，交班时交接 | 流程型需要"交接班记录"功能 |

> **核心决策**：系统通过"工厂类型"配置开关（离散型/流程型/混合型）同时支持两种模式，后台按车间/产线独立配置。

#### M01-2 报工模式体系

根据企业规模和岗位特点，提供三种报工模式：

| 模式 | 定义 | 适用场景 | 工价计算方式 |
|------|------|---------|------------|
| **个人报工** | 每个工人独立报工 | 计件工资、个人绩效清晰的小厂 | 个人产量 × 单件工价 |
| **小组报工** | 班组/产线集体报工，内部分配 | 流水线作业、团队配合 | 小组总产量 × 小组工价 → 内部分配规则（均分/按系数） |
| **设备报工** | 设备自动采集产量，无需人工 | 自动化程度高的产线 | 设备产量 × 设备工价 → 归属到该设备操作工或班组 |

> **报工模式切换**：后台可按车间/产线独立配置，不同产线可采用不同报工模式。

#### M01-7 工价配置体系

**工价类型**：

| 类型 | 定义 | 计算公式 | 适用场景 |
|------|------|---------|---------|
| 计件工资 | 按合格产量 × 单件工价 | 工资 = 合格数量 × 单件工价 | 离散型、个人报工 |
| 计时工资 | 按工时 × 时薪 | 工资 = 报工时 × 时薪 | 设备维护、非计件岗位 |
| 小组计件 | 小组总产量 × 小组工价 → 内部分配 | 先算小组总额，再按规则分配 | 流水线、团队作业 |
| 混合工资 | 底薪 + 计件超额 | 底薪 + (合格数量 - 标准产量) × 超额工价 | 保底+激励 |

**工价配置维度**：

|------|------|------|

| 按工序 | 同一产品不同工序不同工价 | 车削 ¥3/件，铣削 ¥4/件 |
| 按班组 | 不同班组不同工价 | 班组A ¥6/件，班组B ¥5.5/件 |
| 按技能等级 | 高级工更高工价 | 高级工 ¥7/件，初级工 ¥5/件 |
| 按时间段 | 夜班/加班更高工价 | 白班 ¥5/件，夜班 ¥7/件 |
| 按订单类型 | 急单更高工价 | 普通订单 ¥5/件，急单 ¥8/件 |

#### M01-10 生产计划与排程（三级体系）

| 模式 | 定位 | 适用企业 | 核心能力 | 报工反馈 | 闭环机制 |

| **模式B：MRPII 自动计划** | ERP 根据 BOM 和库存自动计算需求 | 物料结构复杂、库存驱动 | 净需求计算、采购建议、生产建议 | 报工更新库存，触发 MRP 重算 | 计划员确认后重跑 MRP |
| **模式C：APS 智能排程** | 算法自动生成最优排程 | 产能紧、交期压力大、多约束 | 有限产能排程、插单重排、换模优化、多目标优化 | 报工偏差超阈值，自动触发 APS 重排 | 系统自动重排 + 人工确认 |

**报工偏差触发 APS 重排的阈值配置**：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| 工时偏差阈值 | 实际工时 vs 标准工时偏差超过 X% | 15% |
| 数量偏差阈值 | 实际产量 vs 计划产量偏差超过 X% | 10% |
| 时间偏差阈值 | 实际完成时间 vs 计划完成时间偏差超过 X 小时 | 2小时 |
| 自动重排开关 | 是否自动触发重排（否则仅提醒） | 关闭（需人工确认） |
| 重排范围 | 仅影响后续任务 / 全量重排 | 仅影响后续任务 |

**APS 算力策略**：
- APS 智能排程属于 Mode C（旗舰版），需要调用 ai.ziwi.cn 算力网关

#### 8.4 IoT 数据完整性保障

> **问题**：采集服务宕机期间，生产继续但数据丢失。制造业数据丢失不可接受，尤其能碳数据涉及合规。

**本地缓存机制**：

```
设备 → IoT网关（本地边缘网关）
  ├── 正常状态：实时上报 → 云端服务 → TimescaleDB
  └── 断线状态：
       ├── 本地缓存（至少 72 小时）
       │   ├── SQLite 本地缓存（网关断电不丢失）
       │   └── 滚动淘汰旧数据
       ├── 恢复后自动补传
       │   ├── 按时间戳顺序重放
       │   └── 云端去重（同设备+同时间戳+同指标 = 同一条数据）
       └── 断线监控
           ├── 本地：LED 指示灯 + 蜂鸣器报警
           └── 云端：断线通知推送到运维人员 OA/短信
```

**数据完整性校验机制**：

| 检查点 | 校验方式 | 处理策略 |
|--------|---------|---------|
| 采集端 | 每条数据带时间戳+设备ID+序列号 | 断线恢复后按序列号补传 |
| 传输层 | MQTT QoS 1（至少一次） | 重复数据由云端去重 |
| 云端接收 | 按设备+时间戳做 UPSERT | 防止重复写入 |
| 日终校验 | 每日凌晨检查各设备采集点数与预期值差距 | 差距超 5% 自动告警 |
| 能碳合规 | 月度数据完整性报告 | 缺数标记为"估算值"，不影响合规

**轻量独立模式（Light Mode）集中定义**（其他章节引用此定义）：

> **轻量独立模式**是指：在客户未选购某项依赖模块的情况下，系统仍可独立运行，但部分数据需通过手动录入或 Excel 导入方式替代。
> **数据兼容性**：轻量模式下的数据格式与标准模式完全一致，客户后期升级到标准模式时**无需数据迁移**，只需开启自动联动的配置开关即可。

| 模式 | 定义 | 替代方式 | 升级路径 |
|------|------|---------|---------|
| **WMS 轻量独立** | 不依赖 ERP 采购/销售单据，独立运行 WMS | 出入库单据在系统内**手动创建** | 开启 ERP 对接 → 原有单据自动关联 M06 |
| **财务轻量独立** | 不依赖 ERP 进销存数据，独立运行财务模块 | 财务数据通过**Excel 导入**或**手动录入** | 开启 M06 对接 → 凭证自动同步 |
| **看板轻量独立** | 无业务模块数据，允许创建空白看板 | 创建空看板 → 后续绑定数据源 | 添加业务模块后自动关联数据 |
| **生产轻量独立** | 无 IoT 数据采集，依赖人工录入 | 产量/工时等数据通过 PC 端手动录入 | 部署 M12 后自动切换为采集数据 |

> **核心原则**：Phase 1 确定定价**逻辑**（计费单位、免费模式、付费边界），定价**数值**留在配置文件中后期运营可改，不上代码逻辑。

#### 5.3.1 Phase 1 定价逻辑

| 决策点 | 选型 | 理由 |
|--------|------|------|
| **计费单位** | 按【已启用模块数】+【活跃用户数】混合 | SaaS 行业最成熟模式，技术方案有现成参考 |
| **免费模式** | 30 天全功能试用 | 最简单的试用期逻辑，无需用量累计 |
| **付费边界** | 所有模块统一付费解锁，无免费模块 | Phase 1 不区分功能分级，简化架构 |
| **试用到期策略** | 限制为只读模式，数据不删除 | 避免数据丢失导致客户流失 |
| **年度折扣** | 年付 8 折（配置可调） | 提高年付比例，降低流失率 |

#### 5.3.2 定价对产品架构的影响

| 定价逻辑 | 架构影响 |
|---------|---------|
| 按模块计费 | 前端菜单动态渲染 + 后端 API 校验模块权限 + 数据表按模块分 schema |
| 按用户计费 | 区分"活跃用户"和"已授权用户" + 用户级权限开关 + 超限自动锁定 |
| 30 天全功能试用 | 试用期管理（注册时间戳） + 到期提醒 + **降级为限量模式** |
| 限量降级 | 每月免费 1,000 次写入操作，超量后提示"本月写入次数已用完，请升级套餐"，读操作不受限制，数据不丢失 |

#### 5.3.6 计费与订单产品规格（评审补充）

|------|---------|

| 不选整月 | 整月方式对客户不友好，按天计算更公平透明 |

| 阶段 | 时间点 | 动作 |

### 7.2 硬件外设对接方案

| 外设 | 对接方式 | 适用终端 | 具体场景 |
|------|---------|---------|---------|
| **条码标签打印机** | **Phase 1 策略**：不做工位打印，做**集中打印**（班组长在办公室 PC 端统一打印标签/合格证后发放到工位，工人报工完成后继续下一个任务，不需等待打印）；**Phase 2 策略**：浏览器打印（CSS@page）+ 专用打印服务（WebSocket → ZPL/EPL） | 触屏终端 → Phase 2 | 派工单标签、物料标签、合格证打印 |

> **注**：车间环境恶劣导致普通打印机寿命短、标签纸规格多、工人不会处理卡纸。Phase 1 不承诺工位打印，集中打印降低风险和成本。硬件需客户自行维护。
| **扫描枪** | USB HID 模拟键盘输入 / WebSocket 直连 | 触屏终端 | 扫工单号、扫物料码 |
| **PDA 扫码** | 调用系统相机解码（ZXing/H5+camera） | PDA | 入库出库扫码 |
| **移动 WAP 扫码** | H5 调用相机扫码（jsQR / QuaggaJS） | 移动 WAP | 抽检扫料号、设备维护扫设备码 |

> **H5 扫码注意事项**：iOS Safari 要求页面必须通过 **HTTPS** 加载才能调用相机 API（`getUserMedia`），且首次使用时需用户**手动授权**相机权限。
> **Phase 1 降级方案**：
> 1. 在扫码按钮旁提供**手动输入码**的备选入口
> 2. 增加**批量扫码**模式：一次扫描/输入多个设备码暂存，统一提交（适合巡检时连续扫多个设备）
> 3. 推荐使用企微/钉钉**内置扫码**（企业微信 `wx.scanQRCode` / 钉钉 `dd.scan`），不受浏览器限制

### 9.3 OA 系统审批与消息对接

> **核心原则**：知微平台不替代企业已有的 OA 系统，而是**融入**企业的日常沟通和审批流，让审批和消息在员工最常用的入口完成。

## 十、能碳系统渐进式整合路线

> **策略调整**：原方案采用"API 互通→数据合并→代码迁移"三阶段。评审后调整为**代码合并 + 功能开关**策略——Phase 1 就开始代码合并，用功能开关控制可见性和功能开放，避免 API 桥接变成硬编码接口耦合，也避免后期数据迁移的停机风险。

### 10.1 整合策略对比

| 维度 | 原方案（API 互通） | 新方案（代码合并 + 功能开关） |
|------|-------------------|---------------------------|
| 架构 | 两套独立系统，API 桥接 | 统一代码库，功能开关控制 |
| 数据 | 两套数据库，Phase 2 迁移 | 统一数据库（新租户），旧系统数据不迁 |
| 风险 | API 桥接硬编码耦合，数据迁移停机 | 无数据迁移，无停机 |
| 历史数据 | 需要迁移 | **不迁移**：历史数据保留在旧系统 |
| 新租户 | 接入旧系统，后期再迁移 | 直接接入统一平台 |
| 旧系统下线 | 需要完全迁移后才能下线 | 不需要下线，新旧并行 |

### 11.2 多租户数据隔离策略

| 隔离级别 | 方案 | 适用场景 |
|---------|------|---------|
| **SaaS 模式** | 共享数据库，租户 ID 字段隔离（tenant_id） | 中小客户，成本优先 |
| **本地部署** | 独立数据库实例 | 大客户，安全优先 |
| **混合** | 核心数据独立库，统计数据汇总库 | 大客户 + 集团版 |

#### 11.2.1 多租户防护设计（防"坏邻居"效应）

> **风险**：共享数据库 + tenant_id 隔离，某个租户的慢查询可能影响所有租户（如月底批量导入 10 万条记录导致全库卡死）。

> **架构师说明**：PostgreSQL 的 `statement_timeout` 是**会话级参数**，无法按 `tenant_id` 原生区分。改用应用层API网关熔断方案替代。

| 防护层 | 措施 | 方案 |
|--------|------|------|
| **查询层** | 慢查询隔离 | **API 网关熔断**（替代 PG statement_timeout）：记录每个租户查询耗时（滑动窗口5分钟P95），超阈值触发该租户写请求熔断（返回429），读请求降级到只读副本 |
| **查询层** | 租户级连接池 | 应用层连接池（SQLAlchemy `QueuePool`），每租户独立连接池（pool_size=5, max_overflow=2）。100租户约需500连接，PG `max_connections` 需调整为500+，或使用PgBouncer事务池（150连接支撑1000租户） |
| **资源层** | 租户级资源配额 | 限制单个租户的 CPU/IO 使用上限（cgroup / RDS 实例级限流） |
| **资源层** | 租户级限流 | API 网关按租户限流（100 req/s 默认，可调整），超限返回 429 |
| **存储层** | 大查询拆分 | 报表类查询走只读副本，不影响主库写入性能 |
| **存储层** | 批量导入限流 | 单次导入不超过 10,000 条，超量自动分片 |
| **备份层** | 租户级备份 | 支持按租户导出数据，可独立恢复特定租户 |
| **监控层** | 租户级告警 | 单个租户 QPS/慢查询/CPU 异常时告警，可快速定位"坏邻居" |

### 11.3 数据库选型

| 用途 | 选型 | 理由 |
|------|------|------|
| 业务数据 | PostgreSQL | 与能碳系统一致，成熟稳定 |
| 时序数据 | TimescaleDB（PG 扩展） | 设备采集、能耗数据 |
| 缓存 | Redis | 会话、消息队列 |
| 文件存储 | MinIO（S3 兼容） | 图片、文档、Excel |

#### 11.3.1 模块化架构技术实现

> **问题**：13 个模块各自可独立部署、升级、关闭，对数据库 schema 设计、服务间降级、前端插件化提出了高要求。

**数据库 Schema 策略**：

```sql
-- 统一数据库，按 module 前缀分 schema
-- 模块启用在应用层控制，数据库不做分库

-- 模块开关表（租户级）
CREATE TABLE tenant_modules (
    tenant_id VARCHAR(50) NOT NULL,
    module_code VARCHAR(20) NOT NULL,   -- M01, M02, ...
    is_enabled BOOLEAN DEFAULT FALSE,
    enabled_at TIMESTAMP,
    PRIMARY KEY (tenant_id, module_code)
);

-- 所有业务表以 module_code 为 schema 前缀命名
-- pg_schema: m01_production, m02_tpm, m03_quality, ...
-- 未启用的模块，其 schema 中的表不会创建索引以避免写开销

-- 查询时校验模块权限（由中间件自动处理）
SELECT * FROM m01_production.work_orders
WHERE tenant_id = ? AND tenant_has_module(?, 'M01');
```

**服务降级策略**：

| 场景 | 处理方式 | 用户体验 |
|------|---------|---------|
| 模块已购买但数据库查询超时 | 返回缓存数据（5 分钟过期） | 看到的是 5 分钟前数据 |
| 模块已购买但服务崩溃 | 显示"模块暂时不可用" + 自动重试 | 短暂中断 |
| 模块未购买 → **API 权限两阶段校验** | 网关层粗筛 + 服务层细筛 | 看不到这个模块 |
| 模块已过期（试用到期） | 菜单显示但只读 + 弹窗提示续费 | 可查看数据但不可操作 |

**API 权限校验两阶段模型**：

```
Stage 1：API 网关层（粗筛，高性能）
┌─────────────────────────────────────────────┐
│  API 网关                                  │
│  请求 → 解析 tenant_id + module             │
│  → 查 Redis（tenant_modules 缓存，TTL 60s） │
│  → 模块未启用 → 直接返回 403                 │
│  → 模块已启用 → 转发到应用服务层              │
│  延迟：< 5ms（Redis 内存查询）               │
└─────────────────────────────────────────────┘

Stage 2：应用服务层（细筛，业务级）
┌─────────────────────────────────────────────┐
│  应用服务（FastAPI Middleware）              │
│  检查用户在该模块内的具体权限：                │
│  · user.role in [admin, operator]           │
│  · 数据范围权限（如只看本车间）               │
│  → 无权限 → 返回 403（含错误码）             │
└─────────────────────────────────────────────┘
```

**前端插件化策略**：

```
┌───────────────────────────────────────────────┐
│              统一前端框架（Vue3）                │
│                                               │
│  ┌──── 菜单渲染层 ───────────────────────────┐ │
│  │  · 从后端 API 获取当前租户已启用的模块列表    │ │
│  │  · 动态渲染菜单项                           │ │
│  │  · 未启用模块的菜单不显示                     │ │
│  └────────────────────────────────────────────┘ │
│                                               │
│  ┌──── 路由懒加载 ───────────────────────────┐ │
│  │  · 每个模块的页面代码独立打包                │ │
│  │  · 未启用的模块路由不注册                    │ │
│  │  · 用户访问未授权路由时显示 404 + 升级引导    │ │
│  └────────────────────────────────────────────┘ │
│                                               │
│  ┌──── 组件共享库 ───────────────────────────┐ │
```

### 17.1 数据访问抽象层（DAL）

> **问题**：SaaS（tenant_id 过滤）和本地部署（无 tenant_id、独立数据库）两套模式下，业务代码需要感知底层差异，容易代码分叉。

**方案**：定义统一 Repository 接口，SaaS 模式和本地部署模式各自实现，业务代码通过依赖注入获取实例。

```python
# Step 1：定义抽象接口
class Repository(ABC):
    @abstractmethod
    def query(self, sql: str, params: dict) -> List[dict]:
        pass

# Step 2：SaaS 模式实现（自动注入 tenant_id）
class MultiTenantRepository(Repository):
    def __init__(self, tenant_id: str):
        self.tenant_id = tenant_id
    def query(self, sql: str, params: dict) -> List[dict]:
        enhanced_sql = f"SELECT * FROM ({sql}) AS sub WHERE tenant_id = :tenant_id"
        params["tenant_id"] = self.tenant_id
        return db_session.execute(enhanced_sql, params).fetchall()

# Step 3：本地部署实现（无 tenant_id）
class SingleTenantRepository(Repository):
    def query(self, sql: str, params: dict) -> List[dict]:
        return db_session.execute(sql, params).fetchall()

# Step 4：依赖注入（FastAPI middleware 根据部署模式注入对应实例）
# 业务代码完全不需要知道是 SaaS 还是本地部署
```

### 17.3 错误码规范

**格式**：`HTTP 状态码 + 4 位业务码`

| 错误码 | 含义 | 场景 |
|--------|------|------|
| `400-1001` | 参数校验失败 | 请求参数格式错误 |
| `400-1002` | 业务规则冲突 | 如出库数 > 库存数 |
| `401-2001` | 认证失败 | Token 无效或过期 |
| `403-2002` | 模块未购买 | 租户未启用该模块 |
| `403-2003` | 用户权限不足 | 角色无该操作权限 |
| `404-3001` | 资源不存在 | 查询的资源 ID 不存在 |
| `409-4001` | 数据冲突 | 重复提交、并发修改 |
| `429-5001` | 租户限流 | 超过 API 调用频率限制 |
| `429-5002` | 租户熔断 | 慢查询超阈值，写请求临时阻断 |
| `500-9001` | 内部服务异常 | 未预期的服务端错误 |
| `503-9002` | 模块服务不可用 | 依赖的模块服务崩溃 |

### 17.9 SaaS + 本地部署数据库连接数估算与风险

| 场景 | 租户数 | 每租户连接 | 总连接 | PG 配置 | 风险 |
|------|--------|-----------|--------|---------|------|
| SaaS 起步 | 10 租户 | 5 | 50 | 默认 100 | 🟢 低 |
| SaaS 中型 | 100 租户 | 5 | 500 | 需调至 500+ 或 PgBouncer | 🟡 中 |
| SaaS 大型 | 500 租户 | 3 (PgBouncer) | 1500 | PgBouncer 事务池 + 分库 | 🔴 高 |

> **PgBouncer 策略**：使用事务级连接池（Transaction pooling），150 个 PG 连接可支撑 1,000 租户。超 1,000 租户时启动分库计划（按租户 ID 哈希分片）。
