# 单元测试规范

> **版本**：v1.0 | **适用范围**：知微 ziwi SaaS 全栈

---

## 一、测试框架与工具

| 层 | 框架 | 断言库 | Mock 库 | 覆盖率工具 |
|----|------|--------|---------|-----------|
| **后端（Python）** | `pytest` | `assert` | `pytest-mock` / `unittest.mock` | `pytest-cov` |
| **前端（Vue 3）** | `Vitest` | `expect`（Vitest 内置） | `vi.mock()`（Vitest 内置） | `@vitest/coverage-v8` |

---

## 二、覆盖率要求

| 层级 | 目标覆盖率 | 说明 |
|------|-----------|------|
| **核心业务逻辑**（Service 层） | **≥ 90%** | 报工计算、审批流转、安灯升级等关键路径 |
| **API 路由层** | **≥ 80%** | 请求/响应格式、参数校验、错误码 |
| **数据访问层**（Repository） | **≥ 70%** | CRUD 操作、租户隔离验证 |
| **项目整体** | **≥ 60%** | 包含工具函数、配置等边缘代码 |
| **前端组件** | **≥ 50%** | 核心交互组件、表单校验 |

> **CI 门禁**：整体覆盖率低于基线时，MR 不允许合并。

## 四、测试编写规范

### 4.1 后端测试（pytest）

#### 基本结构

```python
import pytest
from unittest.mock import AsyncMock, MagicMock


class TestWorkReportService:
    """报工服务单元测试"""

    async def test_submit_work_report_success(self, mocker):
        """正常提交报工应成功更新任务单数量"""
        # Arrange
        mock_repo = mocker.AsyncMock()
        mock_repo.create.return_value = {"id": "wr_001", "quantity": 100}
        service = WorkReportService(repo=mock_repo)

        # Act
        result = await service.submit(
            tenant_id="t_001",
            user_id="u_001",
            data={"order_id": "wo_001", "quantity": 100},
        )

        # Assert
        assert result["id"] == "wr_001"
        mock_repo.create.assert_called_once()

    async def test_submit_exceed_plan_should_reject(self, mocker):
        """提交数量超过计划时应拒绝"""
        # Arrange
        mock_repo = mocker.AsyncMock()
        mock_order_repo = mocker.AsyncMock()
        mock_order_repo.get.return_value = {"id": "wo_001", "planned_qty": 50}

        service = WorkReportService(repo=mock_repo, order_repo=mock_order_repo)

        # Act & Assert
        with pytest.raises(ValueError, match="超过计划数量"):
            await service.submit(
                tenant_id="t_001",
                user_id="u_001",
                data={"order_id": "wo_001", "quantity": 100},
            )
```

#### 命名规范

```
test_{功能}_{场景}_{期望结果}
```

| 部分 | 说明 | 示例 |
|------|------|------|
| 功能 | 测试的功能点 | `submit_work_report` |
| 场景 | 测试的具体场景 | `exceed_plan` |
| 期望结果 | 期望的行为 | `should_reject` |

### 4.2 前端测试（Vitest）

#### 基本结构

```typescript
import { describe, it, expect, vi } from 'vitest'
import { mount } from '@vue/test-utils'
import WorkOrderTable from './WorkOrderTable.vue'

describe('WorkOrderTable', () => {
  it('should render work orders correctly', () => {
    // Arrange
    const orders = [
      { id: 'wo_001', status: '生产中', planned_qty: 100 }
    ]

    // Act
    const wrapper = mount(WorkOrderTable, {
      props: { workOrders: orders }
    })

    // Assert
    expect(wrapper.text()).toContain('wo_001')
    expect(wrapper.text()).toContain('生产中')
  })

  it('should emit "select" when row is clicked', async () => {
    // Arrange
    const orders = [{ id: 'wo_001', status: '生产中', planned_qty: 100 }]
    const wrapper = mount(WorkOrderTable, {
      props: { workOrders: orders }
    })

    // Act
    await wrapper.find('.order-row').trigger('click')

    // Assert
    expect(wrapper.emitted('select')).toBeTruthy()
    expect(wrapper.emitted('select')![0]).toEqual(['wo_001'])
  })
})
```

## 五、Mock 规范

### 5.1 Mock 原则

| 原则 | 说明 |
|------|------|
| **外部依赖必须 Mock** | 数据库、第三方 API、文件系统、MQ |
| **内部逻辑尽量不 Mock** | Service 层方法尽量真实调用（除非有复杂依赖链） |
| **Mock 粒度** | Mock 接口（Repository/Client），不 Mock 具体实现 |
| **避免 Mock 自身代码** | 如果发现需要大量 Mock 自身代码，考虑重构 |

### 5.2 Mock 示例

```python
# ✅ 正确：Mock 外部依赖
mock_repo = mocker.AsyncMock()
mock_repo.get.return_value = {"id": "wo_001", "status": "生产中"}
service = WorkOrderService(repo=mock_repo)

# ❌ 错误：Mock 自身业务方法
mocker.patch.object(service, '_validate_order', return_value=True)
```

## 六、测试数据管理

| 数据来源 | 方式 | 说明 |
|---------|------|------|
| **测试硬编码** | 测试文件内直接定义 | 简单场景（少量数据） |
| **Fixture** | `conftest.py` 中定义 | 多测试共享的数据 |
| **Factory** | `factory_boy`（Python） | 复杂对象构造 |
| **JSON 文件** | `tests/data/*.json` | 大块测试数据（API 响应等） |

### 数据隔离

- 每个测试用例使用独立的数据
- 集成测试使用**独立测试数据库**（`test_ziwi`）
- 测试之间不共享状态

## 七、测试执行

### 7.1 本地执行

```bash
# 后端
cd backend
pytest                                       # 全部测试
pytest tests/unit -v                         # 仅单元测试
pytest tests/integration -v --cov=app        # 集成测试+覆盖率
pytest -k "test_work_report" -v              # 仅报工相关测试

# 前端
cd frontend
npx vitest                                    # 全部测试
npx vitest --coverage                         # 测试+覆盖率
npx vitest src/__tests__/WorkOrderTable.spec.ts  # 单个文件
```

### 7.2 CI 执行

- 每次 PR 自动触发全量测试
- 测试失败阻止合并
- 覆盖率变更在 PR 中自动评论

## 八、测试命名规范速查

| 类型 | 后端 | 前端 |
|------|------|------|
| 正常路径 | `test_{func}_success` | `should work correctly` |
| 异常路径 | `test_{func}_{error}_should_reject` | `should reject when...` |
| 边界值 | `test_{func}_with_max_value` | `should handle max value` |
| 权限 | `test_{func}_without_permission` | `should deny without permission` |
| 租户隔离 | `test_{func}_tenant_isolation` | `should isolate tenant data` |

## 九、禁止行为

| 禁止项 | 说明 |
|--------|------|
| ❌ 测试依赖外部环境 | 测试不应依赖开发/生产环境的 DB 或第三方服务 |
| ❌ 测试间共享状态 | 每个测试独立，不依赖其他测试的执行顺序 |
| ❌ 使用 `time.sleep()` | 异步等待使用 `await asyncio.sleep()` 或 Mock 时间 |
| ❌ 测试数据库真实数据 | 集成测试使用独立测试数据库，不污染开发/生产数据 |
| ❌ 跳过断言 | `pytest.skip()` 应有明确原因和 Issue 跟踪 |
