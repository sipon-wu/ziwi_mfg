### 1.1 数据库 Schema 设计文档

| 内容项 | 说明 |
|--------|------|
| **核心表结构** | 租户表（`tenant`）、用户表（`user`）、权限表（`role`/`permission`/`role_permission`）、模块开关表（`feature_flag`）、工单表（`work_order`）、报工表（`work_report`）、安灯表（`andon_call`）、设备表（`equipment`）、审批表（`approval_template`/`approval_instance`）等 |
| **索引策略** | 哪些字段建索引、联合索引顺序、索引命名规范（`idx_{table}_{column}`） |
| **分区方案** | 按 `tenant_id` HASH 分区，分区数量建议 **16 分区** |
| **时序表设计** | TimescaleDB 超表：chunk 间隔 **1 天**，压缩策略启用，保留策略 **90 天** |
| **主数据归属** | 每类主数据的权威来源模块：物料→M06、设备→M02、产品→M01、人员→M0 |
| **报工数据模型** | 通用字段（`work_report` 主表）+ 扩展属性（`wr_discrete_ext` 扩展表），支持离散型和流程型双模式预埋 |
| **多租户方案** | 所有业务表包含 `tenant_id` 列，Repository 层自动注入，RLS 行级安全策略作为第二道防线 |
| **审计字段** | 所有表包含 `created_at` / `updated_at` / `created_by` / `updated_by` |
| **DDL 脚本** | 提供完整可执行的 `.sql` 脚本，按模块分文件 |

**输出文件**：`知微ziwi-数据库Schema设计文档.md` + `sql/` 目录下的 DDL 脚本

### 2.1 日志规范

**统一日志格式**（JSON，方便日志中心检索）：

```json
{
  "timestamp": "2026-06-12T10:00:00.000Z",
  "level": "INFO",
  "request_id": "req-xxx",
  "tenant_id": "t_12345",
  "user_id": "u_67890",
  "trace_id": "trace-xxx",
  "module": "M01",
  "action": "work_report",
  "message": "报工提交成功",
  "duration_ms": 120,
  "status_code": 200
}
```

| 规范项 | 约定 |
|--------|------|
| 日志级别 | `DEBUG`（开发调试）/ `INFO`（业务操作）/ `WARN`（异常但不中断）/ `ERROR`（需要人工处理） |
| 必填字段 | `timestamp`、`level`、`request_id`、`message` |
| 业务字段 | `tenant_id`、`user_id`、`module`、`action` 尽量填充 |
| 性能字段 | API 请求记录 `duration_ms`，慢查询标记 `WARN` |
| 敏感信息 | 禁止记录密码、Token、身份证号等敏感字段 |

**输出文件**：`知微ziwi-日志规范.md`

### 2.2 Git 分支管理策略

```
main（生产环境）
  │
  └── develop（集成环境）
        │
        ├── feature/xxx（功能分支，从 develop 拉取）
        ├── bugfix/xxx（修复分支）
        └── release/v1.0（发布分支）
```

| 规则 | 说明 |
|------|------|
| 分支命名 | `feature/{模块}-{功能简述}`（如 `feature/M01-work-report`） |
| MR 目标 | feature/bugfix 分支通过 MR 合并到 develop |
| 代码评审 | 至少 **1 人** 通过方可合并 |
| CI 门禁 | 单元测试 + 代码扫描通过后方可合并 |
| 发布流程 | develop → release/vX.X → 打 tag → 合并到 main |
| Commit 规范 | `[模块] 操作: 简述`（如 `[M01] feat: 新增个人报工提交接口`）|

**输出文件**：`知微ziwi-Git分支管理策略.md`

### 2.3 代码规范与 PR 流程

| 内容项 | 说明 |
|--------|------|
| **Python 风格** | 遵循 PEP 8，使用 Black 格式化（line-length=100） |
| **Vue 3 风格** | 遵循 Vue 3 官方风格指南，使用 ESLint + Prettier |

| **PR 模板** | 包含：变更描述、影响范围、测试说明、截图（UI 变更） |
| **Code Review 检查项** | 正确性/安全性/性能/可读性/测试覆盖 |

### 2.4 单元测试规范

| 内容项 | 说明 |
|--------|------|
| **框架选型** | 后端：pytest；前端：Vitest |
| **覆盖率要求** | 核心业务逻辑 **≥ 80%**，整体 **≥ 60%** |

### 2.5 缓存设计文档

| 内容项 | 说明 |
|--------|------|
| **Redis key 命名规范** | `{app}:{module}:{entity}:{id}`（如 `ziwi:M01:work_order:WO20260001`） |
