# 专家组评审输入 — 数据路由与模块整合方案

> **评审范围**：整合设计文档 + 全部历史设计文档的整体回顾
> **评审输入日期**：2026-06-14
> **评审发起人**：齐活林（交付总监）
> **评审成员**：高见远（架构师）、寇豆码（工程师）、严过关（QA）、许清楚（产品经理）

---

## 一、评审背景

近期就"M11 能碳整合 + 自选套餐数据路由"方案进行了三份方案文档的评审及整合。过程中暴露出若干问题，本次专家组评审需针对这些问题逐项审查。

---

## 二、核心评审需求

### 2.1 暴露的问题

| # | 问题 | 描述 | 影响 |
|:-:|:-----|:------|:----|
| P1 | 方案冲突 | M11 能碳方案和路由策略方案在设备查询上用了不同的技术路线（写死 LEFT JOIN vs 条件 if/else） | 架构不统一，后续模块扩展会越来越割裂 |
| P2 | 工作量严重低估 | 源系统迁移自称"约 3h"，实际逐条盘点后约 **28h**（9 倍差距） | 排期不可信，无法制定可靠的项目计划 |
| P3 | FeatureFlag 未落地 | 方案依赖的 FeatureFlag 注入机制还是代码中的 stub，无实际实现 | 整个路由策略无法执行 |
| P4 | 进度反馈缺失 | 评审过程长时间静默，评审人无法感知进展 | 影响协作体验和信任 |
| P5 | 跨文档一致性不足 | 三份方案之间在命名粒度、事务边界、模块耦合等维度存在不一致 | 方案不可信，实施时会有冲突 |

### 2.2 评审目标

1. **严格审查**整合设计文档是否存在遗漏、矛盾、不可实施之处
2. **整体回顾**全部已有设计文档，发现跨文档的一致性问题
3. **可实施性评估**：按现有设计能否直接进入编码阶段？有哪些前置条件不满足？
4. **工作量可信度**：28h 的估算是否合理？是否存在进一步遗漏？
5. **风险发现**：是否有设计层面未覆盖的风险？

## 四、各成员评审重点

### 高见远（架构师）— 架构一致性与完整性审查

**核心检查项**：
1. D1 是否彻底消除了 D2/D3/D4 之间的方案冲突？（特别是设备查询和告警联动）
2. DataRouteResolver 的设计是否足够通用，能支持未来 13 个模块？
3. FeatureFlag JWT 注入方案是否有安全漏洞？
4. 独立部署模式和 SaaS 模式之间的代码复用方案是否合理？
5. **三份旧文档中还有哪些内容未被 D1 覆盖或取舍不当？**
6. 三层路由架构是否存在过度设计？

### 寇豆码（工程师）— 可实施性审查

**核心检查项**：
1. 每个新文件的接口定义是否清晰？能不能直接编码？
2. 现有 Repo 层代码是否需要大规模重构才能适配条件路由？
3. `jose` 库是否已在项目依赖中？是否需要新增依赖？
4. 独立部署的 `standalone_auth.py` 和现有认证系统能否并行运行？
5. **附件的文件变更清单中是否有遗漏的文件？**
6. **实际编码前还有哪些设计细节需要补充？**

### 严过关（QA）— 可测试性与质量审查

**核心检查项**：
1. 条件路由的 32 种场景组合是否都有测试覆盖方案？
2. 参数化测试的设计能否真正覆盖所有分支？
3. 跨模块事务回滚的测试如何设计？
4. 独立部署模式（SQLite 切换）的测试方案是否完整？
5. **现有测试框架是否支持场景驱动的参数化测试？**
6. **最大的测试盲区在哪里？**

### 许清楚（产品经理）— 业务价值与场景审查

**核心检查项**：
1. 6 大客户画像的业务场景是否准确反映了实际客户需求？
2. "独立部署模式"的业务场景是否有明确的客户需求支撑？
3. 供应商模块的归属决策是否影响到了产品路线图？
4. **自选套餐的定价/配置逻辑是否与路由设计方案一致？**
5. 场景规则引擎的约束条件（BLOCK/WARN）是否与产品策略一致？

## 七、背景补充：用户对话中暴露的关键反馈

（供评审成员理解用户关注点）

1. **"应该很快回来" — 用户等不到反馈**：评审过程的透明度需要改进，设计上是否有"进度上报"的机制？
2. **"评审结论我没看到" — 信息未及时同步**：产出物需要更清晰的变更摘要和结论前置
3. **"不是为评审而评审" — 用户希望评审有实际产出**：每条评审意见都必须可追溯、可落地的整改措施
4. **"整体回顾全部设计文档" — 不能只看新文档**：必须检查新旧文档之间的一致性
