## 一、总体判定

### 🔴 不可直接进入编码阶段

**前置条件**：
1. ✅ 设计文档本身的概念是合理的（三层路由分离、声明式路由）
2. ❌ **路由解析器实现存在两个严重 bug（🚫 SUPPLIER 路由策略不可达、死代码）**
3. ❌ **场景驱动的测试基础设施完全未就绪**（`tests/scenarios/` 目录不存在）
4. ❌ **现有 `conftest.py` 不支持按场景切换 feature_flags**
5. ❌ **EnergyRepository 当前无任何条件路由逻辑**（设计中的改造尚未实施）
6. ❌ **独立部署模式的测试方案为空白**

**判定**：设计必须修正路由解析器 bug 后方可开始编码，且测试基础设施需要提前并行准备。

## 二、问题清单

### 🔴 阻断级（Blocking）

| # | 问题 | 类型 | 影响范围 | 详细说明 |
|:-:|:-----|:----|:---------|:---------|
| **B1** | **SUPPLIER 路由策略永远无法匹配** | 设计 Bug | `route_resolver.py:298` | `DataType.SUPPLIER` 的路由规则使用 `("默认", RouteStrategy.EXCEL_IMPORT)` — 这是一个中文字符串 `"默认"`，而 `resolve()` 方法在 L314 检查的是 `flag_or_default == "default"`（英文字符串）。这两个永远不会相等，导致 `ROUTES` 中虽有 SUPPLIER 条目，但 `resolve(DataType.SUPPLIER, ...)` 会进入兜底逻辑并最终返回 `MANUAL_INPUT`，而非预期的 `EXCEL_IMPORT`。 |
| **B2** | **resolve() 中的第二个 for 循环为死代码** | 设计 Bug | `route_resolver.py:319-323` | L316-317 的逻辑是：遍历 route_map，若某项不是 `"default"` 但 feature_flag 匹配，则返回对应策略。如果第一轮遍历中没有匹配任何 flag 也没有命中 `"default"`，控制流才会落到 L319。但此时再遍历一次找 `"default"` 没有任何意义 — 第一轮没命中 `"default"`，第二轮也不可能命中。L323 的 `return RouteStrategy.MANUAL_INPUT` 也是死代码。整个 L319-323 可以安全删除。 |
| **B3** | **场景驱动测试基础设施为零** | 测试缺失 | `tests/scenarios/` | 设计文档第10节详细规划了 `tests/scenarios/conftest.py`、`tests/scenarios/routes/` 下的 3 个测试文件，**但这些文件一个都不存在**。整个 `tests/scenarios/` 目录尚未创建。如需在编码阶段同步交付测试，测试框架需提前搭建。 |

| **B5** | **当前 EnergyRepository 完全不具备条件路由能力** | 实现与设计脱节 | `energy_repo.py` | 当前 `energy_repo.py` 中全部是硬编码的 `SELECT * FROM energy_device` 单表查询，没有任何 TPM JOIN 或路由策略判断。`list_devices`、`get_emission_summary`、`create_alert` 均未使用 `DataRouteResolver`。设计文档第6节的改造尚未实施，测试无法针对路由决策编写。 |
| **B6** | **create_alert 的安灯联动存在事务边界歧义** | 设计不明确 | `energy_repo.py:788-792` | 设计文档 L791 的注释写 `raise  # 或者可以选择不 raise → 仅告警成功`，这是一个"或者"的选择 — 但这是重大行为差异：`raise` = 同事务回滚（告警+安灯都失败），`不 raise` = 告警成功但安灯静默失败。设计文档没有明确选择，这导致测试无法确定预期行为。 |

## 三、补充建议

### 3.1 测试基础设施重构方案

当前 `conftest.py` 需要重构以支持场景测试：

```python
# 建议的方案 — 支持按测试函数切换 feature_flags

# conftest.py 中新增
@pytest.fixture
def feature_flags() -> Dict[str, bool]:
    """默认所有 feature 为 False，测试可覆盖"""
    return {}


@pytest.fixture(autouse=True)
def override_feature_flags(feature_flags):
    """默认覆盖 get_feature_flags 返回空 dict"""
    from app.core.dependencies import get_feature_flags
    async def _mock():
        return feature_flags
    app.dependency_overrides[get_feature_flags] = _mock
    yield
    # NOTE: override_dependencies fixture 会统一清理
```

这样测试可以按需覆盖：

```python
@pytest.mark.parametrize("feature_flags,expected", [
    ({"M02_EQUIPMENT": True}, RouteStrategy.TPM_EQUIPMENT),
    ({}, RouteStrategy.ENERGY_SELF),
])
async def test_device_routing(feature_flags, expected):
    strategy = DataRouteResolver.resolve(DataType.DEVICE, feature_flags)
    assert strategy == expected
```

## 五、最大测试盲区评估

### 🔴 盲区 #1：路由解析器本身不可执行

设计文档中的 `route_resolver.py` 包含两个阻断级 bug（B1+B2），这意味着**即使编写了测试，测试也会因为设计本身的 bug 而失败**。这是 QA 评审中发现的最严重问题 — 设计不可测试，不是测试覆盖的问题，而是设计质量的问题。

### 🔴 盲区 #2：事务回滚无法验证

当前的测试基础设施基于 `unittest.mock.patch` 和 mock `AsyncSession`，这种模式下**无法验证 DB 事务回滚**。因为：
- mock session 的 `execute()` 不执行真正的 SQL
- 没有真正的数据库连接，`rollback()` 是 no-op
- 验证"回滚后数据不存在"需要一个真实的测试数据库

**建议**：为跨模块事务测试引入内存 SQLite（使用 `aiosqlite`），在测试 fixture 中创建隔离的测试数据库。

### 🟡 盲区 #3：FeatureFlag JWT 注入的集成测试

`get_feature_flags` 函数依赖 JWT payload 中的 `features` 字段。但当前的 `get_current_user` 返回的是 `MOCK_USER`（来自 `conftest.py`），没有 `features` 字段。这意味着：
- 单元测试可以 mock `get_feature_flags` 返回值
- 但**端到端测试无法验证 JWT 签发 → 解析 → 注入 → 路由生效**的完整链条

**建议**：增加一个 JWT 集成测试，使用 `create_access_token` 签发含 features 的 token，再用 `verify_token` 解析，验证解析结果与注入数据一致。

### 🟡 盲区 #4：默认值策略的安全影响

设计文档第5.2节定义了 `DEFAULT_FEATURES` 全 False 的安全默认策略。但该函数尚未实现。如果实现有误（例如 `update()` 覆盖了全 False 基底），可能导致未授权模块泄露数据。当前无任何测试覆盖这一安全假设。
