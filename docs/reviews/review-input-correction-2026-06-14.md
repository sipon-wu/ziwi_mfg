## 一、偏差定位

### 文档 v2 的核心前提（错的）

> 能碳是知微云的一个模块，不是两个系统之间的数据同步，而是同一个系统的模块间数据引用。

### 客户（Kane）的实际需求（对的）

> 我的想法还是两个独立系统相互同步。能碳可能独立上，所以需要独立的轻量化输入输出机制。如有其它模块支撑时，就不启用轻量化的输入输出，如独立，就需要。比如用 Excel 表输入，用能碳独立的看板输出展示等。

### 直接后果

| 层面 | 文档走的方向（需修正） | 实际应走的方向 |
|:-----|----------------------|---------------|
| 部署模型 | 所有模块同库/同部署/同认证 | M11 能碳是独立可部署的系统，可单独运行 |
| 数据关系 | `LEFT JOIN` 条件路由，同一 DB 内切换 | 两个系统之间做同步，各自有独立 DB |
| 轻量化 I/O | 未考虑 Excel 导入兜底 | Excel 导入是独立模式的核心能力 |
| 独立看板 | 未涉及 | 能碳独立看板是独立模式的标配 |
| 同步方向 | 同进程 Repo 调用 | 跨系统 API/事件同步 |

## 二、正确认知：三层模型

不是一刀切，而是根据模块特性分三类：

### 类型 A：真独立模块

代表模块：**M11 能碳管理**

特征：
- 可以单独卖给客户，自成业务闭环
- 独立部署，自有数据库（SQLite/PG）、自有认证
- 轻量化 I/O：Excel 导入设备/供应商/能耗数据
- 自有看板：能碳 Dashboard / 大屏
- 有平台支撑时：数据不再从 Excel 来，而是从平台同步过来

### 类型 B：强依赖模块

代表模块：**M05 安灯 → 需要 M02 TPM / M03 品质 → 需要 M01 生产**

特征：
- 不能单独卖，必须有前置模块提供业务上下文（安灯需设备，品质需工单）
- 同平台内部署，同库 JOIN，用条件路由即可
- 不存在"独立系统同步"的问题

### 类型 C：半独立模块

代表模块：**M01 生产管理（单独卖时设备需 Excel 导入）**

特征：
- 可单独卖但数据来源受限
- 单独部署时部分数据需 Excel/手工录入
- 有平台数据基座时自动对接
