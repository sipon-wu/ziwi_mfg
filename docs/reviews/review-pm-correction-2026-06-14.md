#### 偏差 1：类型 B 的"强依赖"定义与实际产品方案矛盾

**评审输入表述**（第70-71行）：
> M05 安灯 → M02 TPM（安灯需要设备上下文）🔴 必须
> M03 品质 → M01 生产（品质检验需要工单/报工）🔴 必须

**产品方案 V1.4 实际表述**（第794-798行依赖矩阵）：

| 依赖关系 | 评审输入等级 | 产品方案 V1.4 等级 |
|:---------|:-----------:|:-----------------:|
| M05 安灯 → M02 TPM | 🔴 必须 | 🔗 建议 |
| M03 品质 → M01 生产 | 🔴 必须 | 🔗 建议 |

**制造业工厂的真实场景分析**：

| 依赖关系 | 有依赖的场景 | 无依赖仍可运行的场景 |
|:---------|:------------|:-------------------|
| **M05 → M02** | 设备故障安灯需读取 M02 的设备台账和状态 | 缺料安灯（关联物料）、质量安灯（关联 M03）、工艺安灯（纯手动输入）、模具更换安灯——**这些场景不需要 TPM** |
| **M03 → M01** | FQC 成品检验需关联 M01 生产工单、IPQC 巡检需关联 M01 工序 | IQC 来料检验（独立于生产）、独立抽检（如出货检验）、**纯品质模块做检验记录管理**——这些场景不需要 M01 |

**制造业实战案例**：
- **纯安灯场景**（A12）：很多电子组装厂只需要一个"拉绳报警"系统，工人按铃→班长回应→记录，**根本不需要设备管理**。安灯的核心价值是"异常响应闭环"，不是"设备联动"。
- **纯品质场景**（A11）：来料检验 IQC 是独立于生产的，小型工厂可能先上线品质模块管 IQC，生产模块后面再上。

**结论**：**评审输入的"必须"过于绝对，建议降级为"建议依赖"**。这与产品方案 V1.4 的商业友好策略一致，也与场景枚举表 A06/A07/A11/A12 中"无 M02 但有 M05"、"无 M01 但有 M03"的场景一致。

## 三、业务场景验证：独立部署场景的真实性评估

### 3.1 M11 能碳独立部署场景 —— ✅ 真实成立

**场景真实性判断**：**完全成立**

**为什么制造业客户可能只需要能碳？**

| 客户画像 | 典型场景 | 驱动因素 | 真实案例参考 |
|:---------|:---------|:---------|:------------|
| **碳合规驱动型** | 出口欧盟企业，需提交碳排放报告 | CBAM（碳边境调节机制）、客户供应链碳审计 | 2026年CBAM全面执行，大量出口制造企业亟需碳管理 |
| **集团能碳管控型** | 集团总部要管各工厂能耗碳排 | 集团 KPI、双碳目标、ESG报告 | 多地工厂的集团型企业 |
| **政府监管驱动型** | 重点用能单位需按时上报能耗数据 | 《重点用能单位节能管理办法》、地方发改委要求 | 年能耗5000吨标煤以上的企业 |
| **已有 MES/ERP 的客户** | 已有生产管理系统，只缺能碳 | 已有系统满足生产管理需求 | 用友/金蝶/Odoo 客户，只买能碳模块 |
| **物业/园区管理** | 非制造业但也需要能碳管理 | 园区能耗管理、碳排放核算 | 工业园区管委会、商业综合体 |

**这些场景的共同特征**：
- 客户不是不需要 MES，而是**已有 MES（或暂时不需要 MES）**
- 能碳需求从"合规"出发，而非从"生产管理"出发
- 购买决策者可能源自**EHS/能源部门**，而非生产部门

### 3.2 轻量化 Excel I/O 在能碳场景下的业务流程

**问题**：Excel 导入在能碳场景下具体是什么频率和流程？

**分析**：能碳场景下的数据导入有明确的行业惯例：

| 数据类型 | 导入频率 | 数据来源 | 说明 |
|:---------|:--------|:---------|:-----|
| **电表读数** | **每日一次**（日更） | 电工每日抄表 → 填入模板 | 大多数中小工厂没有自动集成的电表 |
| **水表/气表/蒸汽表** | 每日一次或每周一次 | 值班人员抄表 | 频率取决于用量和监管要求 |
| **月度能源账单** | **每月一次** | 供电局/燃气公司账单 PDF/Excel | 用于月度核算和合规报告 |
| **设备运行数据** | 每周一次（无 IoT 时） | 设备操作工记录 → 填入模板 | 运行时长、启停次数 |
| **供应商碳数据** | **季度或年度一次** | 供应商填写门户/Excel 模板 | 供应链碳管理的数据采集 |
| **产量数据** | 每日一次 | 生产日报 Excel 导出 → 导入能碳 | 用于碳强度核算 |

**关键发现**：Excel 导入不是"偶尔兜底"，而是**独立模式下能碳模块的数据生命线**。每日一次的电表读数导入是最核心的日常操作。

**建议**：
1. 能碳独立模式下，**日更 Excel 导入**是高频操作，UI 应支持"最近使用的模板"快速入口
2. 提供**导入进度和校验结果**的实时反馈（行数校验、阈值校验）
3. 支持**上次导入数据与本次数据的差异对比**（如"今日用电量较昨日+15%"，提示可能异常）

### 3.3 安灯不依赖 TPM 的独立性验证 —— ✅ 真实成立

**真实场景**：

| 场景 | 安灯触发类型 | 是否需要 TPM |
|:-----|:------------|:------------|
| **缺料呼叫** | 操作工按铃"缺料了" | ❌ 不需要，班长协调领料 |
| **质量异常呼叫** | 质检员发现批量不良，叫停产线 | ❌ 不需要，品质主管处理 |
| **工艺异常呼叫** | 工艺参数超出规格，呼叫工艺员 | ❌ 不需要，工艺员调整 |
| **模具更换呼叫** | 模具寿命到，呼叫换模工 | ❌ 不需要（如果模具单独管理） |
| **设备故障呼叫** | 设备停机，呼叫维修 | ✅ 需要 TPM 获取设备台账和维修记录 |

**结论**：约 **40%~60%** 的安灯呼叫类型与设备无关。纯安灯场景（不关联设备）完全成立。

## 五、改进建议（5 条）

### 建议 1：统一依赖等级标准——从"强制"切换为"建议+轻量独立模式"

**问题**：评审输入中的"🔴 必须"标注与产品方案 V1.4 的"建议非强制"策略、场景枚举表中的实际配置，三者之间存在矛盾。

**具体方案**：

将模块依赖树改为以下标准：

| 等级 | 标记 | 含义 | 配置界面行为 | 举例 |
|:-----|:----|:-----|:------------|:-----|
| **P0 强依赖** | 🔴 | 无此模块系统无法运行 | 自动勾选，不可取消 | 目前没有（M00 除外） |
| **P1 建议依赖** | 🟡 | 有业务关联但有降级方案 | 弹窗提示"建议搭配"，允许继续 | M05→M02, M03→M01, M07→M06 |
| **P2 增强依赖** | 🟢 | 有平台时自动对接，无平台时不影响 | 仅展示提示信息 | M11→M01/M02（有平台增强） |
| **无依赖** | ⚪ | 完全独立 | 无提示 | M11 独立模式 |

**优点**：
- 消除评审输入与产品方案之间的矛盾
- 配置界面不会"卡死"客户（商业友好）
- 为每个依赖提供了明确的降级路径
- 与场景枚举表中 A11/A12/B01 等"无依赖"场景对应

| **Excel I/O 流程** | 有模板定义，但无操作流程 | 补充能碳独立模式的"日更 Excel 导入"UI 流程和快捷操作设计 |
| **同步方向** | 描述为"跨系统 API/事件同步" | 明确同步协议（REST API / Webhook / Event Bus？）、同步频率（实时/准实时/T+1？） |
| **独立看板** | 提到"独立看板是标配" | 独立看板的部署方式（与能碳系统同部署 vs 云端托管 vs SaaS 看板服务？） |
| **数据一致性** | 未涉及 | 两个系统之间的数据对账机制（如何确保两边数据一致？） |

### 建议 4：配置界面交互规则需明确

**问题**：对于"选了 A 模块但缺 B 模块"的场景，配置界面的交互行为需要明确。

**建议规则**：

```
配置校验交互规则（商业友好版）：

规则 1：P0 强依赖（当前不存在，保留占位）
  → 系统自动勾选依赖模块，不可取消
  → 取消时需要先取消当前模块

规则 2：P1 建议依赖（M05→M02, M03→M01, M07→M06 等）
  → 弹窗提示："建议搭配 [模块名] 以获得更好的体验"
  → 提供两个按钮：[了解详情] + [继续配置]
  → 选择"继续配置"后，系统自动为该模块启用"轻量独立模式"
  → 配置单中标注该模块为"轻量模式"

规则 3：P2 增强依赖（M11→M01/M02）
  → 展示信息提示："搭配 [模块名] 可自动同步数据"
  → 不阻止配置继续

规则 4：混合选择（如"能碳+安灯"但不选 TPM）
  → 能碳 → 无依赖，独立运行（Excel 导入能耗数据）
  → 安灯 → 建议搭配 M02（弹窗提示），允许继续（启用轻量模式，手动选择关联对象）
  → 系统不报错，配置单标注两个模块均为"轻量模式"
```

**视觉原型建议**：

```
[M05 安灯]  ⚠ 轻量模式
建议搭配 M02 TPM 以启用"设备故障自动关联安灯"功能。
[了解轻量模式]  |  [无需，继续配置]
```

### 建议 5：补充评审输入中"类型 A 独立系统"的演进路径说明

**问题**：M11 当前作为类型 A（真独立），但产品方案的整合路线明确它是**渐进式融入平台**的——Phase 1 SSO 集成，Phase 2 数据层统一，Phase 3 代码合并。这意味着 M11 的"类型"是随时间变化的。

**建议**：在评审输入中增加"类型演进矩阵"：

| 阶段 | M11 的"类型" | 数据架构 | 部署方式 |
|:-----|:------------|:---------|:---------|
| **Phase 1（Now）** | 类型 A（真独立） | 独立 DB，API 同步 | 独立部署或与平台同部署 |
| **Phase 2（Next）** | 类型 A→C 过渡（半独立） | 共享 M12 数据层，独立认证 | 仍可独立，但推荐集成 |
| **Phase 3（Later）** | 类型 C（半独立→完全整合） | 统一数据库，统一认证 | 代码完全合并，但可选独立部署 |

**另外**：建议在评审输入中增加**M11 的"双模式 Feature Flag"设计**：

```
feature_flag: energy_carbon_deployment_mode
  - standalone: 独立模式（Excel I/O + 独立看板 + 独立认证）
  - integrated: 平台集成模式（数据从 M01/M02/M12 同步 + SSO）
  
当 standalone=True 时：
  - 显示"Excel 导入"菜单和"独立看板"入口
  - 数据来源为 Excel/手动录入
  - 不调用 M01/M02/M12 的数据接口
  
当 standalone=False 时：
  - 隐藏 Excel 导入入口（数据从平台同步）
  - 使用平台统一的看板模块
  - 自动从 M01/M02/M12 获取数据
```
