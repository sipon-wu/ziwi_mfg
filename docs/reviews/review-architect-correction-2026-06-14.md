## 一、同步架构方案建议

### 1.1 三种方案对比

#### 方案 1：API 拉取（Sync Pull）

| 维度 | 评估 |
|:-----|:-----|
| **设计** | M11 能碳系统定时/按需调用知微云平台的 REST API，拉取设备/供应商/产量等数据 |
| **优点** | 实现最简单；能碳系统无需感知平台内部结构；接口可复用现有 API |
| **缺点** | 实时性差（依赖轮询间隔）；接口耦合度高（API 变更影响 M11）；缺乏背压机制 |
| **适用场景** | 低频数据（供应商、产量汇总）；已有 API 可复用的短期过渡方案 |
| **数据一致性** | 弱一致（轮询间隔内数据不一致） |
| **复杂度** | ⭐（最低） |

#### 方案 2：事件推送（Event Push）

| 维度 | 评估 |
|:-----|:-----|
| **设计** | 知微云平台数据变更时，通过消息队列（RabbitMQ / Redis Stream / 本地事件表）推送事件，M11 能碳系统消费 |
| **优点** | 实时性好（秒级推送）；松耦合（M11 不依赖平台 API 版本）；天然的背压和重试机制 |
| **缺点** | 需要消息中间件基础设施；事件丢失的场景需要兜底拉取（T+1 对账） |
| **适用场景** | 高频数据（设备变更、IoT 采集数据）；需要实时同步的场景 |
| **数据一致性** | 最终一致（事件+兜底拉取保障） |
| **复杂度** | ⭐⭐⭐（中等） |

#### 方案 3：共享数据源（Shared DB View）

| 维度 | 评估 |
|:-----|:-----|
| **设计** | M11 能碳系统直接连接知微云数据库的只读视图/只读账号 |
| **优点** | 数据一致性最强（实时一致）；无需开发同步接口；查询灵活 |
| **缺点** | **严重违反独立系统原则**：跨 DB 访问引入强耦合；安全风险（数据库暴露给外部系统）；网络依赖导致 M11 不可离线运行；无法支持 SQLite 独立部署 |
| **适用场景** | 几乎不适用（仅在企业内部局域网且信任边界明确的场景可能考虑） |
| **数据一致性** | 强一致（实时 DB 查询） |
| **复杂度** | ⭐⭐（开发简单但运维复杂） |

### 1.2 推荐方案：Event Push + API Pull 混合（Hybrid Sync）

```
┌──────────────────────┐         ┌──────────────────────┐
│    知微云平台         │         │  M11 能碳独立系统     │
│                      │         │                      │
│  ┌────────────────┐  │  Event  │  ┌────────────────┐  │
│  │ M01 生产管理   │──┼─────────┼─▶│ 事件消费服务    │  │
│  │ M02 TPM 设备  │  │  Push   │  │ (Event Consumer)│  │
│  │ M05 安灯      │  │         │  └────────┬───────┘  │
│  └────────────────┘  │         │           │          │
│         │            │         │           ▼          │
│         ▼            │         │  ┌────────────────┐  │
│  ┌────────────────┐  │         │  │ M11 本地 DB    │  │
│  │ 事件总线        │  │         │  │ (SQLite/PG)    │  │
│  │ (Event Bus)    │──┼───API───┼─▶│                │  │
│  │ Redis Stream   │  │  Pull   │  │ energy_device  │  │
│  │ 或 本地事件表   │  │ 兜底    │  │ carbon_record  │  │
│  └────────────────┘  │         │  │ energy_alert   │  │
│                      │         │  └────────────────┘  │
└──────────────────────┘         │                      │
                                 │  ┌────────────────┐  │
                                 │  │ Excel 导入     │  │
                                 │  │ (独立模式数据源)│  │
                                 │  └────────────────┘  │
                                 └──────────────────────┘
```

#### 推荐理由

| 理由 | 说明 |
|:-----|:------|
| **符合独立系统原则** | M11 保持独立数据库和部署，不直接访问平台 DB |
| **实时性满足业务需求** | 设备变更/产量报工 → 秒级同步到能碳 |
| **支持离线运行** | M11 独立系统可脱网运行（本地 SQLite + Excel 导入），联网后自动同步 |
| **兜底机制保障数据一致** | Event Push 作为主通道，定时 API Pull 作为兜底对账 |
| **渐进式复杂度** | Phase 1 可用"本地事件表"（无消息中间件），Phase 2 升级为 Redis Stream/消息队列 |

#### Phase 1 轻量实现方案（无消息中间件）

```
知微云平台                          M11 能碳系统
──────────                          ────────────
数据变更
    │
    ▼
写入本地 change_log 表              HTTP POST /api/v1/sync/pull
(记录: table, row_id, action, ts)  (M11 定时轮询, 默认 60s)
    │                                    │
    └─── 等待 M11 拉取 ──────────────────▶
                                         │
                                         ▼
                                    M11 解析 change_log
                                    → 调用平台 API 获取完整数据
                                    → 写入本地 DB

定时对账(每日 02:00):
  平台: SELECT count(*) FROM equipment WHERE updated_at > last_sync
  M11: 对比本地数据量 → 不一致时全量拉取
```

### 1.3 数据路由设计的范围修正

| 类型 | 数据架构 | 数据路由适用性 |
|:-----|:---------|:-------------|
| **A 类：真独立模块（M11）** | 独立 DB，通过 API/Event 同步 | ❌ DataRouteResolver **不适用**跨系统场景。数据来自 Excel 导入或 API 同步，不涉及同库 JOIN |
| **B 类：强依赖模块（M05→M02, M03→M01）** | 同库部署 | ✅ DataRouteResolver **适用**同平台条件路由 |
| **C 类：半独立模块（M01/M06/M07/M10）** | 同库为主，轻量模式可降级 | ✅ DataRouteResolver **适用**同平台场景；轻量模式对应 `MANUAL_INPUT`/`EXCEL_IMPORT` 策略 |
| **D 类：基础设施（M12/M13）** | 同库，横向服务 | ✅ DataRouteResolver **适用**（如 M12 数据来源路由） |

### 2.2 逐文件详细评估

#### ✅ `route_resolver.py` — 保留，但缩小范围

**现有功能**：基于 `(DataType, feature_flags) → RouteStrategy` 的声明式路由解析。

| 评估维度 | 结论 |
|:---------|:-----|
| **设计质量** | 良好。声明式路由优于 if/else 分支，扩展性佳 |
| **适用场景（修正后）** | ❌ 不用于 M11 跨系统同步（独立系统间不存在同库 JOIN）<br>✅ 用于同平台模块（B 类：M05→M02 安灯联动路由；C 类：M01 轻量模式降级路由；D 类：M12/M13 数据来源路由） |
| **需要修改** | 在类注释/文档中明确适用范围：`此路由解析器仅用于同平台模块（B/C/D 类），不适用于跨独立系统同步` |
| **DataType 枚举调整** | `DataType.SUPPLIER` → 保留（M01 共享供应商的同平台路由）<br>其余 DataType 保留（DEVICE/ALERT/EMISSION/PRODUCTION/MONITOR 用于同平台场景） |

#### ✅ `dependencies.py` — 保留

**现有功能**：`get_feature_flags` 从 JWT 解析 features，`get_tenant_repo` 注入 Repo。

| 评估维度 | 结论 |
|:---------|:-----|
| **适用场景** | 知微云平台端。同平台模块通过 FeatureFlag 控制路由 |
| **对 M11 独立系统的影响** | ❌ M11 独立系统不使用此注入链（M11 使用自己的认证和 Repo）<br>✅ M11 **作为平台模块时**（即全平台模式），仍走此注入链 |
| **需要修改** | 无。但需增加注释明确：M11 独立模式下不经过此流程 |

#### ✅ `security.py` — 保留

| 评估维度 | 结论 |
|:---------|:-----|
| **features 参数** | 有效。JWT 携带套餐信息是合理的跨请求传递方式 |
| **密钥统一** | 已验证统一为 `JWT_SECRET_KEY`，符合 v1.1 设计 |
| **需要修改** | 无 |

#### ✅ `auth_service.py` — 保留

| 评估维度 | 结论 |
|:---------|:-----|
| **login 写入 features** | 逻辑正确。平台用户登录时从 `Tenant.package_modules` 转换为 feature_flags |
| **需要修改** | 建议增加 `features` 字段在 login 返回值中透传给前端，方便前端缓存（目前未返回） |

#### ⚠️ `energy_repo.py` — 需要部分重构

**核心问题**：当前实现的 `list_devices` 通过 DataRouteResolver 走同一 DB 的 JOIN 或自查。但在独立系统架构下：

| 场景 | 数据来源 | 当前代码是否适用 |
|:-----|:---------|:---------------|
| **M11 独立部署（B01）** | Excel 导入 → M11 本地 DB | ⚠️ 当前 `_list_devices_standalone_mode`（自查 energy_device 表）逻辑仍在，但数据来源是 Excel 导入，不是空的 |
| **M11 + 平台（C04）** | 平台 M02 equipment 表 → 通过同步层到 M11 | ❌ 当前代码是 `equipment JOIN energy_device` 同库查询。修正后应该走：平台 equipment 变更 → Event Push → M11 消费 → 写入 M11 本地 energy_device |
| **M11 + 平台 + 独立看板** | 混合：平台数据通过同步层，能耗数据 M11 自有 | ❌ 当前代码无法区分"同库直接查"和"从同步层消费" |

**重构方向**：

```
修正前 (同库 JOIN):
  list_devices():
    if has TPM:  SELECT FROM equipment JOIN energy_device  (同库)
    else:        SELECT FROM energy_device                   (同库)

修正后 (独立系统同步):
  list_devices():
    if STANDALONE_MODE:
        # 数据来源于 Excel 导入 或 Event Push 写入
        SELECT FROM energy_device  (M11 本地 DB)
    else:
        # 同平台模式：直接从平台 DB 查 (保留原逻辑)
        if has TPM:  SELECT FROM equipment JOIN energy_device
        else:        SELECT FROM energy_device
```

**实际上更简单的做法**：在 M11 独立模式下，**所有数据都从 M11 本地 DB 读取**，只是数据填充方式不同（Excel 导入 vs 事件同步）。不需要 DataRouteResolver 判断 JOIN 路径。

```
重构后 (统一从 M11 本地 DB 读):
  list_devices():
    # 无论数据来源是 Excel 导入还是事件同步
    # 数据都已经在 M11 本地 DB 的 energy_device 表中
    SELECT FROM energy_device WHERE ...
    # 不需要管数据是怎么来的（由同步层负责）
```

**这意味着**：`energy_repo.py` 中的路由选择代码大部分可以简化，核心变为**"只从 M11 本地 DB 读取"**。数据写入路径才是需要根据模式切换的：

```
create_device():
  if STANDALONE_MODE:
      # 写入 M11 本地 energy_device 表
      INSERT INTO energy_device ...
  else:
      # 平台模式：写入平台 equipment 表（让平台处理）
      # 通过平台 API 创建设备 → 平台 Event Push 回来 → M11 消费
      await self._platform_api.create_equipment(data)
```

#### `base.py` — 保留

MultiTenantRepository 的 feature_flags 注入对平台端 Repo 仍有价值，无需修改。

## 三、数据一致性方案

### 3.1 同步方向与数据主权

基于**"谁创建谁主责"**原则定义数据主权：

| 数据类型 | 数据主权方 | M11 角色 | 同步方向 |
|:---------|:----------|:---------|:---------|
| **设备档案** | **M02 TPM**（平台设备台账是唯一权威来源） | 🔴 **只读消费** | M02 → M11（单向） |
| **供应商主数据** | **M01 生产管理**（供应商主数据在 M01） | 🔴 **只读消费** | M01 → M11（单向） |
| **产量数据（报工）** | **M01 生产管理** | 🔴 **只读消费** | M01 → M11（单向） |
| **能耗数据** | **M11 能碳**（能源数据由能碳系统产生） | 🟢 **数据所有者** | M11 自有，不同步到平台（除非平台需要） |
| **碳排放核算结果** | **M11 能碳** | 🟢 **数据所有者** | M11 自有 |
| **能耗告警** | **M11 能碳** | 🟢 **数据所有者** | M11 自有（告警可联动触发平台安灯——单向） |
| **IoT 时序数据** | **M12 IoT 网关** | 🟡 **半消费** | M12 → M11（若 M12 存在且连接） |

**核心原则**：
- **设备/供应商/产量**：M11 是"数据消费者"，平台是"数据生产者"→ **单向同步，平台覆盖 M11**
- **能耗/碳排放**：M11 是"数据所有者"→ **不同步回平台**（除非平台需要做全局能耗分析）
- **告警联动**：M11 可触发平台安灯（M05），但这是"操作"而非"数据同步"

### 3.2 冲突策略

| 场景 | 冲突类型 | 解决策略 |
|:-----|:---------|:---------|
| 平台修改了设备名称，M11 也修改了同一设备名称 | 双向修改冲突 | **平台覆盖 M11**（设备主权在平台） |
| 平台删除了设备，M11 有该设备的能耗记录 | 引用完整性冲突 | **软删除策略**：平台标记删除 → 同步到 M11 标记为"已停用"（保留历史能耗数据） |
| M11 通过 Excel 导入的设备，平台没有 | 数据孤岛 | **不自动反向同步**。Excel 导入的设备仅存在于 M11。若后来开通平台，需做数据归并 |
| 平台和 M11 同时新增了设备（不同编码） | 重复创建 | **以平台编码为准**：M11 新增设备时不与平台共用 ID 空间，通过 `sync_id` 和 `source` 字段区分来源 |

### 3.3 同步频率

| 数据类型 | 主通道 | 频率 | 兜底对账 |
|:---------|:-------|:-----|:---------|
| **设备变更** | Event Push（实时） | 秒级 | T+1 全量对账 |
| **供应商变更** | Event Push（实时） | 秒级 | T+1 全量对账 |
| **产量报工** | Event Push（准实时） | 分钟级 | T+1 全量对账 |
| **能耗数据** | 无（M11 自有） | — | — |
| **IoT 数据** | Event Push（若 M12 存在） | 秒级 | T+1 全量对账 |

### 3.4 M11 本地 DB 的数据模型补充

为支持多来源数据，`energy_device` 表需增加来源跟踪字段：

```sql
-- M11 本地 energy_device 表补充字段
ALTER TABLE energy_device ADD COLUMN data_source VARCHAR(20) DEFAULT 'excel';
  -- 'excel' = Excel 导入（独立模式）
  -- 'sync'  = 从平台同步（全平台模式）
  -- 'manual' = 手动录入

ALTER TABLE energy_device ADD COLUMN source_id VARCHAR(64);
  -- 平台端对应记录的 ID（如 equipment.id），用于同步关联

ALTER TABLE energy_device ADD COLUMN sync_version INTEGER DEFAULT 0;
  -- 乐观锁，用于检测同步冲突
```

## 四、I/O 机制设计建议

### 4.1 统一导入模板规范

Excel 导入是 M11 独立模式的数据生命线（日更操作），必须标准化。

#### 模板格式规范

```
┌─────────────────────────────────────────────────────────────┐
│  知微 SaaS — 数据导入模板 V1.0                               │
│  (第一行: 固定表头 + 元数据)                                  │
├─────────────────────────────────────────────────────────────┤
│  模板编码: ZIWI_IMP_001                                      │
│  模块: M11 能碳管理                                           │
│  数据类型: 设备导入                                           │
│  版本: 1.0                                                   │
├─────────────────────────────────────────────────────────────┤
│  device_code* │ device_name* │ device_type │ energy_type │ ... │
```

### 4.2 标准化导入接口

所有模块共用统一的导入 API 接口，避免每个模块各自实现一套：

```python
# POST /api/v1/import/upload
# 统一的导入入口
{
    "template_id": "ZIWI_IMP_001",      # 模板 ID
    "file": "<Excel 文件>",
    "mode": "insert" | "upsert" | "replace",  # 导入模式
    "options": {
        "skip_header": true,
        "batch_size": 500,
        "on_error": "skip" | "abort"
    }
}

# 返回结果（统一格式）
{
    "code": 0,
    "data": {
        "import_id": "imp_20260614_001",
        "status": "completed",       # completed / partial / failed
        "total_rows": 1500,
        "success_rows": 1498,
        "error_rows": 2,
        "errors": [
            {"row": 23, "field": "device_code", "message": "设备编码格式错误"},
            {"row": 456, "field": "energy_type", "message": "无效的能源类型"}
        ],
        "warnings": [
            {"row": 89, "message": "设备编码 DEV-001 已存在，已跳过"}
        ]
    }
}
```

### 4.3 三层校验规则

导入校验分三层执行，每一层的错误阻断后续层：

```
Layer 1: 模板格式校验 (File Level)
  ├── 文件是否是有效的 .xlsx/.xls
  ├── 模板编码是否匹配
  ├── 表头列名是否与模板定义一致
  └── 必填列是否缺失
        ↓ 通过
Layer 2: 字段级校验 (Field Level)
  ├── 数据类型校验（数值/字符串/日期）
  ├── 格式校验（正则表达式）
  ├── 枚举值校验（能源类型是否在允许列表）
  ├── 长度校验
  └── 唯一性校验（如设备编码在本次导入中不重复）
        ↓ 通过
Layer 3: 业务逻辑校验 (Business Level)
  ├── 设备编码是否与现有数据冲突（DB 查重）
  ├── 关联数据存在性校验（如引用的设备 ID 是否存在）
  ├── 阈值校验（如能耗值是否在合理范围内）
  └── 业务规则校验（如结束时间 >= 开始时间）
        ↓ 通过
写入数据库
```

### 4.4 独立看板的设计建议

PM 评审中提到的"独立看板"建议采用**同部署同数据源**策略：

| 方案 | 描述 | 评价 |
|:-----|:-----|:-----|
| **A. 内嵌看板（推荐）** | 看板组件与 M11 系统同部署，直接从 M11 本地 DB 查询 | ✅ 最简单，离线可用，数据权限清晰 |
| **B. 云端看板服务** | 看板渲染放在知微云平台，M11 推数据到平台 | ❌ 增加复杂度，违背独立系统原则 |
| **C. 嵌入式 BI 工具** | 集成 Metabase/Superset 等开源 BI | ⏳ 可远期考虑（Phase 3） |

**Phase 1 建议采用方案 A**：M11 前端直接渲染看板，数据来自 M11 本地 DB。

## 六、风险矩阵

### 6.1 新方案的主要技术风险

| # | 风险描述 | 等级 | 概率 | 影响 | 缓解措施 |
|:-:|:---------|:----:|:----:|:-----|:---------|
| R1 | **事件遗漏导致数据不一致**：平台 change_log 记录丢失或 M11 消费失败，导致两边设备/供应商数据不一致 | 🔴 高 | 中 | M11 核算碳排放时使用了错误的产量/设备数据，影响合规报告 | Phase 1 用"拉取模式"（M11 主动拉）+ 重试机制；Phase 2 增加 T+1 全量对账 + 告警通知 |
| R2 | **Excel 导入数据质量问题**：用户手工填写的 Excel 格式错误、数据异常，导致 M11 独立模式下数据不可用 | 🔴 高 | 高 | 能碳模块无法正常运行（日更电表读数失败） | 三层层校验（模板→字段→业务）；导入前预览（行数+校验结果）；导入失败保留原始文件供排查 |
| R3 | **同步延迟导致实时数据不可用**：Event Push 延迟或 M11 拉取周期过长，看板展示的数据不是最新的 | 🟡 中 | 中 | 独立看板数据滞后，用户体验下降 | 默认拉取周期 60s（可配置）；看板标注"数据更新于 X 分钟前"；关键指标（如当前功率）走 IoT 直连 |
| R4 | **双向同步环路**：M11 写了数据 → 平台 event → M11 又消费回来，导致数据重复或死循环 | 🟡 中 | 低 | 数据重复写入 | 明确"单向同步"原则（设备/供应商只从平台到 M11）；本地数据标注 `data_source=sync` 和 `source_id`，消费时做幂等判断 |
| R5 | **M11 离线模式下积压事件过多**：M11 断网数天，重新联网后需消费大量积压事件 | 🟡 中 | 中 | 同步耗时过长，CPU/内存飙升 | 分页拉取（默认 100 条/次）；消费进度持久化（记录 last_sync_id）；全量对账替代逐条重放 |
| R6 | **独立部署的 SQLite 并发写入冲突**：Excel 导入（写）和事件同步（写）同时操作同一表 | 🟡 中 | 低 | SQLite 写入锁冲突导致超时 | SQLite WAL 模式（支持读写并发）；导入操作放在低峰期（如夜间）；后续可升级为 PostgreSQL |
| R7 | **模板版本不兼容**：平台升级了导入模板格式，M11 独立部署使用的是旧版模板 | 🟡 中 | 中 | Excel 导入大面积失败 | 模板版本管理（SemVer）；向后兼容策略（v1.0 模板始终可用）；导入时校验模板版本，不兼容时提示用户下载新版 |
| R8 | **T+1 对账发现不一致但无法自动修复**：全量对账发现 1000 条数据不一致，需要人工判断 | 🟢 低 | 中 | 对账报告太长无人处理 | 自动修复规则（以平台数据为准覆盖 M11）；无法自动修复的标记为"异常"并告警；提供差异对比界面供人工确认 |

## 附录：核心不等式

```
修正前（错误认知）:
  M11 = 知微云平台的一个模块
  → 同库 JOIN / 条件路由 / FeatureFlag / 函数调用
  → DataRouteResolver 万事通

修正后（正确认知）:
  M11 ≠ 知微云平台的一个模块
  M11 = 能碳独立系统 ↔⟨Event Push + API Pull⟩↔ 知微云平台

  同平台模块（B/C/D 类）:
    → 同库 JOIN / 条件路由 / FeatureFlag / 函数调用 ✅

  跨系统同步（A 类 M11 ↔ 平台）:
    → API 同步 / 事件同步 / Excel 导入 ❌ 不同库
    → DataRouteResolver 在这里无用 ❌
```
