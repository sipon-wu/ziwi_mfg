# 产品经理评审报告 — 数据路由与模块整合方案

> **评审人**：许清楚（产品经理）
> **评审日期**：2026-06-14
> **评审对象**：`data-routing-integrated-design.md` v1.0
> **视角**：业务价值与场景覆盖

---

## 总体判定

**设计方案在技术架构层面方向正确，但从产品与业务视角看存在 3 个阻断级问题，不建议在当前状态下进入编码阶段。**

### 可进入编码的条件

必须满足以下前置条件后方可启动编码：
1. ✅ **供应商模块归属决策完成**（涉及 P2 迁移方案是否重做）
2. ✅ **独立部署模式的产品定位正式确认**（技术方案因定位不同差异巨大）
3. ✅ **自选套餐变更的端到端体验流程完成产品设计**（不仅是后端 Token 刷新）

## 一、🔴 阻断级问题

### 🔴-1：供应商模块归属未决策，P2 迁移方案可能全盘重做

**出处**：设计文档 §9.2 供应商模块说明 + §4.2 DataType.SUPPLIER 路由规则

**问题描述**：
设计文档标注 supplier.py（5 个端点，3h）为"需决策归属 M11 专用还是全平台通用供应商模块"。但当前 `DataRouteResolver.ROUTES` 中已将 `DataType.SUPPLIER` 硬编码为 `EXCEL_IMPORT`（所有场景默认 Excel 导入）。如果最终决策供应商是"全平台通用模块"，则当前路由方案完全不适用——全平台供应商应有自己的 CRUD 和数据库表，路由策略应是"共享供应商表"而非"Excel 导入"。

**业务影响**：
- P2 迁移的 3h 工时估计不可信——如果供应商模块需要重做路由，工作量至少翻倍
- 影响 Phase 2 排期：供应商归属决策未做出的情况下，P2 的"迁移供应商"任务无法排入 Sprint
- 客户影响：能碳客户如果同时需要供应商管理功能，当前设计只给 Excel 导入，完全不能满足业务需求

**整改建议**：
1. 在进入 Phase 1 编码前完成供应商模块归属决策（建议本周内产品+架构决策会议）
2. 在 ROUTES 表中明确标注 SUPPLIER 的路由方案为"待决策"而非隐式默认
3. 将供应商决策作为 Phase 1 的前置依赖任务加入实施路线图

---

### 🔴-2：独立部署模式的产品定位模糊导致技术方案不可信

**出处**：设计文档 §7 独立部署模式设计

**问题描述**：
当前独立部署模式的技术方案（SQLite + 内置管理员账户 + 硬编码默认密码）暗示这是一个"轻量演示版"，但设计文档的用语是"独立部署模式"（正式的产品形态）。二者存在严重矛盾：

| 维度 | 当前设计 | 如果是最小可行版(MVP) | 如果是正式产品版 |
|:-----|:---------|:---------------------|:----------------|
| 数据库 | SQLite | SQLite 可接受 | 需 PostgreSQL 或至少 SQLite WAL 模式+备份策略 |
| 认证 | 单管理员+硬编码密码 | 可接受 | 需至少 SSO/LDAP 或多管理员 |
| 多租户 | 无 | 合理（单租户） | 取决于定价策略 |
| 升级路径 | 无 | 需设计 | 需设计 |
| 数据迁移至全平台 | 无 | 需设计 | 必须设计 |

**业务影响**：
- 如果"独立部署"是要正式销售的产品，当前方案完全不满足企业客户的安全合规要求
- 如果"独立部署"只是 PoC/演示版，则架构复杂度可接受，但应明确标注"不适用于生产环境"
- 缺乏从独立部署升级到全平台的客户旅程设计——购买了独立版的客户未来如果扩展模块，数据如何平滑迁移？

**整改建议**：
1. **立即决策**：独立部署模式的产品定位是什么？A) 正式销售的单租户版？B) 演示/PoC 版？C) 离线版（面向无网络工厂）？
2. 根据定位重新评估技术方案，重点关注：认证安全性、数据备份恢复、版本升级路径
3. 产品侧补充"独立部署→全平台"的客户升级旅程文档

---

### 🔴-3：自选套餐变更的端到端体验在产品设计层面缺失

**出处**：设计文档 §3.3 JWT 刷新端点

**问题描述**：
套餐变更的流程在设计文档中只覆盖了后端端点（`POST /refresh-features`），但完整的端到端体验涉及：

```
套餐变更 (管理员操作)
  → 系统记录新套餐
  → 触发 JWT 刷新
  → 前端收到新 Token
  → UI 更新模块可见性
  → 用户感知变化
```

当前缺失的关键环节：
1. **时序问题**：套餐变更后立即刷新 JWT 还是用户下次登录时生效？产品策略不明确
2. **前端主动刷新机制**：前端如何知道套餐已变更？需要 WebSocket 推送还是轮询？
3. **降级场景的数据处理**：客户从"制造+能碳"降级为"仅能碳"——已有的 M02 联动数据如何处理？产品层面没有兜底策略
4. **Token 生命周期**：当前 JWT 过期时间 30 分钟（来自 config.py），套餐变更后旧的 30 分钟窗口期内旧 Token 仍有效——这是产品可以接受的吗？

**业务影响**：
- 直接面向客户的功能（套餐变更后的 UI 反馈）完全没有设计
- 如果客户在套餐变更后看不到预期变化，会产生大量客服工单
- 降级场景缺乏数据策略可能导致数据孤岛或客户数据丢失风险

**整改建议**：
1. 产出一份"套餐变更体验流程"文档，定义：生效时机、通知方式、前端刷新机制、降级数据策略
2. 考虑引入 WebSocket 推送或短轮询机制，使前端能及时感知套餐变化
3. 针对降级场景，设计"数据保留但隐藏"的策略（非删除），允许客户在未来升级时恢复

## 二、🟡 重要问题

### 🟡-1：6 大画像在路由设计中实质性缺位

**出处**：设计文档 §4 DataRouteResolver + §10.4 测试场景

**问题描述**：
设计文档在 §10.4 测试策略中引用了 6 大客户画像（A-E），但在路由核心设计（DataRouteResolver.ROUTES）中完全没有体现"画像"这个抽象层次。路由规则是纯 feature_flag 级别的 if/else，丢失了"画像级别的组合策略"。

举例：画像 C（制造+能碳，购买了 M01+M02+M05+M11）和画像 B（仅能碳，只有 M11）在设备列表上走不同路径，但路由表中无法表达"当客户属于画像 C 时，设备创建应该自动创建 equipment + energy_device 联动"——这条规则需要在多个 Repo 方法中重复实现。

**业务影响**：
- 新增套餐组合时，需要手动验证 ROUTES 表中的每条规则是否正确——没有"按画像验证"的机制
- 业务层面的"套餐类型"与技术层面的"flag 组合"之间缺乏映射关系，产品需求到技术实现的链路断裂
- 测试策略中的画像测试（§10.4）与路由规则定义（§4）脱节，测试可能覆盖不全

**整改建议**：
1. 在 DataRouteResolver 中引入"画像模板"层：`persona → [flags组合, 路由策略模板]`
2. 或至少在文档中补充"画像 ↔ Flag 组合 ↔ 路由策略"的映射矩阵
3. 将测试策略中的画像场景前移到设计文档的核心部分，而非仅在测试章节出现

---

### 🟡-2：场景规则引擎过于单薄，不足以支撑 13 个模块

**出处**：设计文档 §11 场景规则引擎

**问题描述**：
当前场景规则引擎只定义了 3 条约束（1 条 BLOCK + 2 条 WARN），对于一个计划扩展到 13 个模块的 SaaS 平台严重不足。以下约束在当前设计中完全缺失：

| 缺失的约束 | 业务逻辑 | 建议等级 |
|:-----------|:---------|:---------|
| M05 安灯需要 M02 设备 | 安灯呼叫必须关联设备 | BLOCK |
| M11 能碳实时监测需要 M12 IoT | 无 IoT 网关时实时监测不可用 | WARN（功能降级显示） |
| M11 能效分析需要 M12 数据采集 | 能效分析依赖时序数据 | WARN |
| M11 碳足迹需要 M01 生产 | 产品碳足迹需要产量数据 | WARN |
| M03 品质检测需要 M02 设备 | 品质检测需关联检测设备 | BLOCK |
| M02 保养计划需要 M02 设备 | 保养计划依赖于设备档案 | BLOCK |

**业务影响**：
- 客户可能配置出功能不可用的套餐组合（如买了安灯但没买设备）
- 产生售后投诉："我买了安灯模块为什么用不了？"
- 随着 Phase 2 模块扩展，约束规则需要持续补充，当前框架的规则表达能力有限（条件表达式仅为字符串）

**整改建议**：
1. 立即扩展约束规则到至少 10 条（覆盖当前 7 个模块间的主要依赖关系）
2. 约束规则不只是后端校验，前端套餐配置页面也应集成同样的校验逻辑
3. 设计"功能降级"策略：当依赖模块未购买时，有些功能应降级显示而非完全不可见

### 🟡-3：28h 迁移工作量中的产品决策风险被低估

**出处**：设计文档 §9.4 迁移优先级原则

**问题描述**：
28h 工作量中，system.py 的"碳资产功能（配额/交易/CCER）"被标注为"需判断是否确实属于 M11，还是应提取为独立的 Mxx 模块"。这条决策的延迟风险被低估：

- 如果最终决策"碳资产提取为独立模块 Mxx"→ 当前的 0.5h 估计作废，需要从零开始设计新模块的路由策略
- 碳资产涉及配额、交易、CCER 等业务逻辑，与当前 M11 能碳核算的定位有关联但不完全相同
- 如果在 Phase 1 完成后再做模块归属决策，可能导致 Migration 方案返工

**业务影响**：
- 碳资产管理是企业碳合规的核心场景（特别是出口欧盟的 CBAM 需求），延迟决策可能影响产品上市时间
- 如果归入 M11，M11 的产品边界会变得过于宽泛（能效+碳核算+碳交易），与产品路线图"模块化"的初衷矛盾

**整改建议**：
1. 在 Phase 1 Sprint 启动前完成碳资产模块归属决策
2. 建议产品侧明确 M11 的产品边界：是"能碳管理"还是"双碳管理"？前者侧重能效，后者包含碳资产交易
3. 如果决定拆分为独立模块，建议提前注册 M14（或 Mxx）的 FeatureFlag 命名

### 🟡-4：FeatureFlag 默认假策略与存量客户模块扩展的冲突

**出处**：设计文档 §5.2 默认值策略

**问题描述**：
`DEFAULT_FEATURES` 将所有已知模块默认置为 False，且 `get_feature_flags` 函数中将 JWT 中未携带的 flag 也视为 False。这意味着：

- 当 Phase 2 新增 M04/M06/M07 等模块时，所有存量客户的 JWT 中不含新 Flag → 自动视为 False
- 存量客户即使购买了新模块，也需要手动触发 `/refresh-features` 才能生效
- 但产品层面没有通知机制告知客户/管理员"您的套餐已包含新模块，请刷新"

**业务影响**：
- 功能开通的延迟会导致客户投诉和客服工单
- 灰度发布时无法精细控制哪些客户先看到新模块
- 与"自选套餐"的理念矛盾——客户自主选配的模块应该即时可用

**整改建议**：
1. 制定"新增模块的灰度发布策略"：是自动对所有已购买客户生效？还是需要管理员确认？
2. 设计"客户端 Token 自动刷新"机制（如在 JWT 过期前检测到 feature_flags 变化时主动刷新）
3. 或考虑将 feature_flags 改为"许可清单"模式（只列出已购买的模块），未在清单中的模块自动不可用

## 三、🟢 建议

### 🟢-1：建议补充"画像驱动的路由模板"

当前 DataRouteResolver 是基于单个 FeatureFlag 的线性判断。建议引入"画像模板"机制：

```
画像模板 C（制造+能碳）:
  设备 → TPM_EQUIPMENT（自动 equipment JOIN energy_device）
  告警 → ANDON_LINKED（自动联动安灯）
  产量 → MES_AUTO（从 M01 工单获取）
  碳排放 → MES_AUTO（关联 M01 产量数据）
  
画像模板 B（仅能碳）:
  设备 → ENERGY_SELF（仅 energy_device）
  告警 → ENERGY_ALERT_ONLY（仅内部告警）
  产量 → MANUAL_INPUT（手动录入）
  碳排放 → MANUAL_INPUT（手动录入能耗数据）
```

这样产品新增套餐组合时，只需"选择画像模板"而非"逐条配置路由规则"。

### 🟢-2：建议将场景规则引擎前端化

当前 ScenarioEngine 是纯后端校验。建议在套餐配置 UI 中集成同样的规则校验，实现"选择即校验"：

- 管理员配置套餐时，前端实时显示依赖检查结果
- 选择 M05 但未选 M02 → 立即显示 🔴 BLOCK 提示
- 提升管理员配置体验，减少"提交后才知道不行"的情况

### 🟢-3：建议补充"模块间依赖矩阵"正式文档

当前模块间依赖散落在场景规则引擎的 3 条 YAML 规则中。建议产出一份正式的"模块依赖矩阵"（表格式），按三个级别标注：

| 依赖模块 | 被依赖模块 | 依赖强度 | 说明 |
|:---------|:-----------|:---------|:------|
| M05 安灯 | M02 设备 | 🔴 强依赖 | 安灯需关联设备 |
| M11 能碳 | M02 设备 | 🟡 建议依赖 | 无设备时能碳功能受限 |
| M11 能碳 | M12 IoT | 🟡 建议依赖 | 实时监测需要 IoT |
| M13 看板 | 任一业务模块 | 🔴 强依赖 | 无数据源时无法展示 |
| ... | ... | ... | ... |

此矩阵同时是产品定价和套餐设计的重要参考依据。

### 🟢-4：建议补充跨模块事务边界的"超时兜底"策略

§8 中定义了告警联动安灯的同一事务模式，代码中有 `logger.warning` 但 raise 了异常导致事务回滚。建议 Product 层面明确：
- 安灯联动失败时，告警应保留还是回滚？
- 从业务角度看，**告警是核心操作，安灯联动是增值操作**。应该安灯失败不阻断告警创建（走独立事务或补偿逻辑）

### 🟢-5：前端独立构建的体验一致性

§7.4 中独立部署使用 `vite.config.standalone.ts` 构建独立前端。建议关注：
- 独立版与全平台版的 UI 一致性维护成本（两套构建配置）
- 独立版的功能降级显示策略——哪些按钮/菜单需要隐藏？
- 建议统一采用"运行时 Flag 控制 UI 元素显隐"而非两套构建

## 四、场景覆盖分析

### 4.1 覆盖矩阵

| 画像 | 场景编码 | feature_flags | 设计中的路由 | 覆盖评估 |
|:-----|:---------|:--------------|:-------------|:---------|
| A-纯制造 | A04 | M01+M02+M03+M05 | ✅ 设备→TPM，告警→安灯联动 | ✅ 完全覆盖 |
| B-能碳驱动 | B01 | 仅M11 | ✅ 设备→自管，告警→内部 | ⚠️ 独立部署场景需补充安全方案 |
| B-能碳+设备 | B04 | M02+M11+M12 | ✅ 设备→TPM，IoT采集 | ✅ 完全覆盖 |
| C-制造+能碳 | C04/C05 | M01+M02+M05+M11(±M03) | ✅ 推荐套餐路由路径清晰 | ✅ 完全覆盖 |
| D-数据采集 | D01 | 仅M12 | ✅ 仅采集模式 | ✅ 完全覆盖 |
| E-看板增强 | C04+M13 | M01+M02+M05+M11+M13 | ⚠️ 看板数据聚合路由略模糊 | 🟡 需补充看板数据源路由细节 |

### 4.2 覆盖发现

1. **画像 E（看板增强）的路由设计最薄弱**：看板需要从多个模块聚合数据，文档中只提到"看板不能独立存在"的约束，但看板本身在 DataRouteResolver 中没有对应的 DataType。看板是一个跨模块视图，不是数据源，其路由逻辑应是"根据已购模块动态展示数据"——这需要比当前 DataRouteResolver 更上层的聚合逻辑。

2. **缺少 F 画像（供应链协同）**：文档提到 6 大客户画像，但现有设计只覆盖了 A-E（5 个）。第六个画像很可能涉及 M04（物料/仓储）或 M06/M07 等 Phase 2 模块。虽然 Phase 2 尚在规划中，但建议在 ROUTES 表中预留相关 DataType 的占位。

3. **所有画像的"数据导入"场景都依赖 M12**：当前设计对 M12 Excel 导入的依赖是硬性的（R5 风险已识别）。如果 M12 未就位，所有画像的"数据导入"功能都会受影响。建议确认 M12 的交付时间线与 Phase 1 对齐。

### 4.3 遗漏场景

| 遗漏场景 | 影响画像 | 说明 |
|:---------|:---------|:------|
| 多工厂/多场地数据隔离 | B/C | 企业客户有多个工厂时，能碳数据是按工厂独立管理还是汇总？ |
| 设备退役/替换的生命周期 | A/C | energy_device 关联的 equipment 退役后，能碳历史数据如何处理？ |
| 能耗告警的升级机制 | B/C | 告警未处理 → 通知升级（短信/邮件/安灯）的流程未在任何画像中定义 |
| 碳核算的第三方审计导出 | B/C | 企业碳核算结果需要导出为标准格式（如 ISO 14064）供第三方审计 |
