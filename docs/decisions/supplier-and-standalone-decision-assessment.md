## 2. 待决议题一：供应商模块归属

### 2.1 问题陈述

供应商模块（supplier.py，5 个端点，P2 迁移预估 3h）在数据路由中归属哪个域？

- **方案 A（当前默认）**：归 M02（生产管理 / TPM）
  - 路由判定依赖 `M02_EQUIPMENT` flag
  - 供应商作为设备供应商（备件、维保）管理
- **方案 B（待决策）**：归 M01（基础主数据 / 报工管理）
  - 路由判定依赖 `M01_WORK_ORDER` flag
  - 供应商作为生产基础数据（原料、物料供应商）管理

### 2.2 方案对比

| 评估维度 | 方案 A：归 M02（生产管理） | 方案 B：归 M01（基础主数据） |
|:---------|:--------------------------|:----------------------------|
| **数据生命周期** | 创建→采购关联→维保→淘汰，生命周期在 M02 内部闭环 | 创建→跨模块共享→多业务环节使用→淘汰，生命周期跨模块 |
| **跨模块复用性** | ❌ 低。M03 品质（来料检验）如需供应商数据需跨模块调用 M02，M11 供应链碳足迹也需跨模块 | ✅ 高。M01 作为主数据源，M02/M03/M05/M11 均可直接引用 |
| **路由复杂度** | ✅ 低。单模块路由，路由表简单 | 🟡 中。需考虑"哪些模块拥有供应商访问权限"的路由策略 |
| **业务语义** | 供应商 = 设备/服务的提供方。语义较窄，割裂了供应商作为企业基础实体的属性 | 供应商 = 企业的业务伙伴（无论提供原料还是设备）。语义完整，符合行业惯例 |
| **数据冗余风险** | M03/M05 若各自维护供应商，会产生数据冗余和不一致 | 一份主数据，所有模块引用同一来源，天然消除冗余 |
| **对 P2 迁移的影响** | supplier.py 纳入 M02 批次，与设备管理耦合 | supplier.py 纳入 M01 扩展批次，作为独立的数据迁移任务 |
| **与竞品对标** | ❌ 不符。SAP（BP 概念）、Oracle（Supplier 独立模块）、用友/金蝶（基础档案）均将供应商作为独立/基础主数据管理，不嵌入执行模块 | ✅ 符合行业主流做法 |
| **对 M11 碳足迹的支持** | 需跨模块调用 M02 获取供应商数据 → 增加路由复杂度和事务边界 | M01 直接暴露供应商数据 → M11 通过 M01 报工关联即可获取 |

### 2.3 行业对标

| 系统 | 供应商归属 | 说明 |
|:-----|:-----------|:------|
| SAP S/4HANA | **BP（Business Partner，基础主数据）** | 供应商/客户统一为 BP，独立于采购/销售模块 |
| Oracle EBS | **PO（采购模块，但 Supplier Master 独立维护）** | 供应商主数据在采购模块维护，但各模块共享引用 |
| 用友 U8+ | **基础设置 → 客商档案** | 供应商作为基础档案管理，不归属任何业务模块 |
| 金蝶 K/3 WISE | **基础资料 → 核算项目** | 供应商作为公共基础资料 |
| 西门子 Opcenter | **Foundational Data Layer** | 供应商作为工厂模型的一部分，跨模块共享 |

**小结**：行业主流做法——供应商是**基础主数据（Master Data）**，而非某个执行模块的内部数据。

### 2.4 推荐方案

**推荐方案 B（归 M01 基础主数据）**，理由如下：

1. **业务实体完整性**：供应商是独立的企业实体（名称、统一社会信用代码、资质、评级），不是"设备的一个属性"。同一个供应商可能同时提供设备和原料，归入 M02 会丢失这种业务的完整性。

2. **跨模块复用是刚需**：知微平台的定位是 13 个模块全链路覆盖。M03 品质检验的来料检验需要供应商数据判定不合格率趋势；M05 安灯可能因原料问题触发，需要追溯供应商；M11 供应链碳足迹必须知道每个供应商的碳排放数据。这些场景全部要求供应商数据跨模块可达。

3. **路由成本可控**：方案 B 的路由策略可以设计为"供应商主数据仅归 M01 管理（CRUD），其他模块走读权限"。当前 DataRouteResolver 的 ROUTES 表中，`DataType.SUPPLIER` 只需增加一条 `M01_WORK_ORDER` 的判定，其余模块需要供应商数据时通过 Repo 组合获取即可。

4. **技术修订量可控**：将当前 ROUTES 中的 `DataType.SUPPLIER` 从 `EXCEL_IMPORT` 改为共享供应商表 + M01 路由即可。不会导致 P2 迁移方案全盘重做（PM 评审的担忧可以解除）。

### 2.5 采纳方案 B 后的 ROUTES 修订建议

```python
DataType.SUPPLIER: [
    ("M01_WORK_ORDER", RouteStrategy.SHARED_SUPPLIER),   # M01 共享供应商表
    ("default", RouteStrategy.EXCEL_IMPORT),              # 无 M01 → Excel 导入降级
],
```

需新增 `RouteStrategy.SHARED_SUPPLIER` 策略枚举值，对应"共享供应商表（M01 主数据）"的分支逻辑。

## 3. 待决议题二：独立部署模式产品定位

### 3.1 问题陈述

M11 能碳模块独立部署场景下，产品定位有三种选项：

| 选项 | 定位 | 当前设计匹配度 | 架构影响 |
|:-----|:------|:--------------|:---------|
| **A（Phase 1 默认）** | 演示版/体验版（PoC / Sales Demo） | ✅ 基本匹配（SQLite + 内置管理员） | 低 |
| **B** | 正式单租户版（可销售的生产级产品） | ❌ 不匹配（需 PostgreSQL、SSO/LDAP、备份恢复） | 高 |
| **C** | 离线版/内网版（无网络依赖） | ❌ 不匹配（需数据同步、冲突解决机制） | 最高 |

### 3.2 方案对比

| 评估维度 | 选项 A：演示版 | 选项 B：正式单租户 | 选项 C：离线版 |
|:---------|:--------------|:------------------|:--------------|
| **目标用户** | 销售演示、PoC 试用的潜在客户 | 仅购买能碳模块的 SMB 制造业客户 | 无网络 / 涉密工厂 |
| **可销售性** | ❌ 不可销售。不能用于生产环境 | ✅ 可销售，可作为独立 SKU | ✅ 可销售，适用特定场景 |
| **功能边界** | 功能受限（演示级别数据量，认证简化） | 与 SaaS 全功能一致，仅数据隔离 | 与 SaaS 功能一致，但无实时 IoT 能力 |
| **技术投入** | ✅ 低（~4h，当前方案） | 🟡 中（~12-16h，需补充 PostgreSQL 支持、SSO、备份策略） | 🔴 高（~25-30h，需离线同步引擎） |
| **升级路径** | ❌ 缺失。Demo 数据无法平滑迁移到正式环境 | ✅ 清晰。数据可直接迁移至全平台 | 🟡 复杂。需数据合并与冲突解决 |
| **市场验证价值** | ✅ 可快速获取产品反馈、验证市场需求 | 🟡 需产品已进入成熟期才适合投入 | 🟡 小众需求 |

### 3.3 风险分析

#### 选项 A 风险

| 风险 | 等级 | 说明 |
|:-----|:----:|:------|
| 客户期望错配 | 🔴 | 演示版标注不清时，客户可能误解为正式产品，导致商务纠纷 |
| 升级路径缺失导致客户流失 | 🟡 | 试用客户满意后无法平滑转正式→可能流失到竞品 |
| 安全合规风险 | 🔴 | 硬编码密码+SQLite 无法通过企业安全审计 |

#### 选项 B 风险

| 风险 | 等级 | 说明 |
|:-----|:----:|:------|
| 过度投资 | 🟡 | 市场尚未验证独立部署的真实需求规模 |
| 与 SaaS 版可能产生功能分歧 | 🟡 | 两套产品线维护成本高 |
| 定价冲突 | 🟡 | 单租户版定价若低于 SaaS 等效套餐，可能加速 SaaS 客户流失到本地部署 |

### 3.4 推荐方案

**推荐分阶段策略：Phase 1 走选项 A（演示版），Phase 2 根据验证结果升级至选项 B（正式单租户版）。**

#### Phase 1（当前 Sprint）—— 选项 A：演示版/体验版

**产品命名**：建议使用"能碳管理体验版"（Energy Carbon Lite）而非"独立部署版"，明确传达"非生产级"定位。

**实施要点**：
- 接受当前的技术方案（SQLite + 内置管理员）
- 同时必须增加以下轻量补充（投入 ~2-3h）：
  1. **启动时打印醒目的免责声明**：`⚠ 体验版仅用于功能演示和 PoC 验证，不建议用于生产环境`
  2. **内置一个"升级咨询"入口**：客户体验后可通过表单提交需求，自动通知销售团队跟进
  3. **在管理界面标注"体验版"水印或标识**
- 认证安全下限：密码从环境变量读取（已实现），但需增加密码复杂度提示

**Phase 1 交付标准**：
- 销售团队可用 Demo 环境做 PoC
- 客户可在 14 天内免费试用（通过计时器控制）
- 试用结束后引导升级咨询

#### Phase 2（市场验证后）—— 升级至选项 B：正式单租户版

**触发条件**（满足任一即可启动）：
1. 体验版累计获得 ≥10 个有效 P0/P1 商机
2. 累计 ≥3 个客户明确表示"体验版功能满足需求，但要求生产级安全合规"
3. 产品路线图将"能碳单租户版"列为正式 SKU

**升级内容**：
- 数据库：SQLite → PostgreSQL 可选（保留 SQLite 作为轻量选择）
- 认证：内置管理员 → 至少支持 OAuth2 / LDAP 集成
- 运维：增加一键备份/恢复功能
- 升级路径：设计"体验版→正式版"的数据迁移工具
- 定价：建议按设备接入数量或数据量定价（与 SaaS 订阅价形成互补）

#### 选项 C（离线版）—— 暂不实施

理由：目标用户群体过小（无网络/涉密工厂），市场验证成本高。建议作为 Phase 3+ 路线图选项，在正式单租户版有稳定客户群后再评估。

## 4. 待业主确认的问题清单

以下问题需 Kane 确认后，两个议题的决策方可最终锁定：

### Q1（供应商）：供应商在知微的业务实体模型中，是否会被 M02（设备供应商）、M03（来料检验追溯供应商）、M11（供应链碳足迹追溯供应商）**多处需要**？

**背景**：如果供应商数据只在 M02 内部使用（仅设备维保场景），方案 A 够用。如果有或计划有跨模块共享需求（品质溯源、碳足迹），方案 B 是唯一正确的选择。

**建议方向**：从"13 个模块全链路覆盖"的产品定位看，跨模块共享是必然，建议选方案 B。

### Q2（独立部署）：独立部署的获客目标是什么？A) 作为销售漏斗的顶部——体验后引导购买 SaaS 版？B) 作为一个独立创收的 SKU，面向有数据隔离需求的客户？

**背景**：这个问题的答案直接影响 Phase 1 的实施策略和投资力度。
- 若回答 A → 选项 A（演示版）足够，重点放在"升级转化漏斗"
- 若回答 B → 需要将 Phase 2 的选项 B 提前到 Phase 1.5，投资升级路径设计

### Q3（独立部署）：是否有已经签约或高意向的客户明确表示需要独立部署？若有，他们的核心诉求是"数据隔离"还是"无网络"还是"成本低于 SaaS"？

**背景**：如果已有真实客户需求，对定位的选择影响巨大。数据隔离→选项 B，无网络→选项 C，成本低→选项 A 的上游需求会直接转化到选项 B。

## 5. 补充决策：供应商主数据归属 & M09 SRM 定位修订

### 5.1 Kane 最终确认

Kane（客户决策者）于 2026-06-14 确认：

- **供应商归属 → 方案 B（M01 基础主数据）**，理由：
  - 供应商如同 BOM，是基础数据而非某个执行模块的属性
  - 采购功能在 ERP 中管理，知微不涉及采购交易逻辑
  - 供应商与采购（PO/合同/价格）是强关联，但在知微内部仅需供应商档案供各模块引用

### 5.2 企业边界澄清

```
客户有 ERP（金蝶/用友/SAP/Odoo 等）
  ┌─ ERP ──────────────────────────┐
  │  供应商主数据（创建/维护/编码）     │
  │  采购管理（PO/合同/收货/对账）     │
  └────────┬────────────────────────┘
           │ 数据同步（M06 桥接层）
           ▼
  ┌─ 知微 ──────────────────────────┐
  │  M01 引用/缓存供应商档案          │
  │  M09 SRM 准入/评价/协同层         │
  │  M03/M05/M11 引用供应商数据       │
  └──────────────────────────────────┘

客户无 ERP
  ┌─ 知微 ──────────────────────────┐
  │  M01 供应商主数据（CRUD）         │
  │  M09 SRM 供应商协同               │
  │  M06 ERP 轻量采购管理             │
  └──────────────────────────────────┘
```

### 5.3 主数据权威来源修订

原产品方案 V1.4 第 1829 行定义：

| 主数据 | 旧权威来源 | 新权威来源（修订） |
|:-------|:----------|:------------------|
| 供应商主数据 | M09 SRM（无SRM时为 M06 ERP） | **ERP 同步**（有ERP）→ **M01**（无ERP）→ M09/M06/M11 消费 |

### 5.4 M09 SRM 定位修订

| 维度 | 原定义 | 修订定义 |
|:-----|:-------|:--------|
| **核心定位** | 供应商主数据管理 + 协同 | **供应商全生命周期管理**（准入→评价→协同→门户） |
| **主数据责任** | 权威来源 | ❌ **免责**。档案数据来自 ERP 同步或 M01 |
| **增值功能** | — | 供应商准入评审、多维评分、自动评级、供应商自助门户 |
| **独立运行性** | 需要 M06 | ✅ 可独立（档案来自 M01 或 Excel 导入） |
| **对 ERP 的依赖** | 不强 | ✅ 强。有 ERP 时做协同层；无 ERP 时做轻量采购辅助 |

### 5.5 影响范围

| 影响项 | 说明 |
|:-------|:-----|
| **数据路由表** | `DataType.SUPPLIER` → `RouteStrategy.SHARED_SUPPLIER`，无误 |
| **M09 SRM 设计** | 需剥离"供应商主数据管理"职责，聚焦准入/评价/协同/门户 |
| **M06 ERP 设计** | 桥接层定位更清晰：同步 ERP 供应商数据到 M01，无 ERP 时提供轻量采购管理 |
| **产品方案 V1.4** | 需修订主数据归属表（§7.2）和 M09 SRM 模块说明（§5.9） |
| **数据路由设计文档** | 需确认 ROUTES 表与修订后的定位一致 |
