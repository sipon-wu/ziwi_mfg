## 第十三部分：第 105 条专项评估 — 实验室与试产合并分析

### 评审意见
第 105 条（SAP/鼎捷）：实验室管理与试产管理是否应该合并为一个模块？

### 产品决策：不合并

| 维度 | 合并方案 | 分离方案（当前设计） | 决策 |
|:----|:--------|:------------------|:----:|
| **职责定位** | 试产包含实验室功能 | 试产=产品验证，实验室=中立检测服务 | ✅ 分离 |
| **委托来源** | 仅试产 | 试产/品质/客诉/手动四类来源 | ✅ 分离 |
| **数据隔离** | 实验室数据与试产数据耦合 | 独立数据表，各自隔离 | ✅ 分离 |
| **复用性** | 品质模块无法使用实验室 | 品质、试产均可发起实验委托 | ✅ 分离 |
| **角色体系** | 实验室角色混入试产 | 四视图角色（实验员/需求方/主管/计量员）独立 | ✅ 分离 |
| **计量管理** | 无独立计量功能 | 计量员视图（内校/外校/计量记录） | ✅ 分离 |

**结论**：实验室是中立的检测服务提供方，与试产的管理目标和角色体系完全不同。合并会降低实验室的复用价值。维持 M15（实验室）+ M16（试产）独立模块设计。

## 第十四部分：第 114 条 — UI 设计 3-5-8 原则响应

### 评审意见
SAP 专家组指出 UI 设计缺少统一原则，建议参考 3-5-8 设计原则（3 次点击可达目标功能/5 秒页面加载完成/8 秒操作响应反馈）。

### 处理方式
1. 新建 `docs/ui-design-guidelines.md` 设计规范文档，包含：
   - **3 次点击原则**：核心生产操作（报工/安灯呼叫/检验判定）≤3 次点击完成
   - **5 秒加载原则**：列表页/详情页 SSR 首屏加载 ≤5 秒，数据聚合看板 ≤3 秒
   - **8 秒响应原则**：事务型操作（创建/提交/审批）后端响应 ≤8 秒（P95 分位）
2. 三端适配规范：
   - **大屏端**（M19 综合看板）：信息密度高/自动轮播/≥42 寸
   - **PC 端**（业务操作）：功能完整/鼠标交互/键鼠快捷键
   - **PDA 端**（仓储现场）：大按钮（≥44px）/扫码优先/三步操作范式

## 第十五部分：二期可扩展域专项说明（EHS/TMS/CRM/HR/财务/SRM）

### 决策调整

根据产品负责人评审确认，F07-F12（EHS/TMS/CRM/HR/财务/SRM）六个域**不是永久放弃**，而是**预留为二期可扩展域**——基于知微云平台二开后部署给有需要的客户。当前 Phase 1 做好可扩展设计，确保未来无缝衔接。

### 各域扩展预留方案

| 域 | 当前 Phase 1 预留设计 | 未来扩展路径 | 二开接入方式 |
|:--|:---------------------|:------------|:------------|
| **EHS**（环境健康安全） | M13 能碳管理预留环境指标扩展字段（水质/废气/噪声）；Integration Gateway 预置 EHS 数据接收接口 | 二期扩展为完整 EHS 子模块（环境监测/职业健康/安全管理） | 标准 API 数据接入 + 模块插件化 |
| **TMS**（运输管理） | M20 出库管理预留发货物流字段（承运商/运单号/发货时间）；Integration Gateway 预置 TMS 集成适配器模板 | 二期扩展为运输管理模块（运单/调度/签收/运费结算） | 适配器模式（自定义 TMS 协议适配器） |
| **CRM**（客户管理） | M10 品质管理预留客户投诉处理入口；Integration Gateway 预置 CRM 客户同步接口 | 二期扩展为轻量 CRM（客户台账/商机/售后/投诉闭环） | Webhook 双向同步 + 标准 API |
| **HR**（人事管理） | M00 人员数据预留扩展属性字段（组织架构自定义属性）；Integration Gateway 预置 HR 组织同步接口 | 二期扩展为 HR 子模块（考勤/培训/资质/排班） | 数据同步插件（AD/OA 适配器） |
| **财务（成本核算）** | M08 报工数据预留单价/金额字段；Integration Gateway 预置财务数据推送 Webhook | 二期扩展为轻量成本核算（工序级成本/工单成本/期间结算） | 数据推送 + 外部财务系统调用 |
| **SRM**（供应商管理） | M10 IQC 检验数据预留供应商接口输出；Integration Gateway 预置 SRM 双向适配器模板 | 二期扩展为供应商门户（准入/评分/改进/协同） | 标准 API + 门户子域名扩展 |

### 架构原则

详见架构师输出的 `docs/extensibility-design.md` 文档，核心原则：

1. **模块插件化**：二期域的代码以独立插件模块形式存在，不侵入 Phase 1 核心代码
2. **扩展点+钩子模式**：在关键业务节点预留扩展点（工单完成→触发成本核算钩子；IQC 完成→触发 SRM 评分钩子）
3. **Integration Gateway 统一入口**：所有二期域的外部数据交换通过 Integration Gateway，不直连
4. **数据模型预留**：核心表预留通用扩展字段（JSONB/自定义属性），二期无需改表结构
5. **二开不碰核心**：通过标准 API + 插件机制实现，禁止直接修改核心表结构
