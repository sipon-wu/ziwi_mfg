## 1. 产品目标 & 用户故事

### 1.1 一句话目标

> **为 mfg1.ziwi.cn 预发布环境构建一套"数据量达标、场景真实、演示可用"的规模工厂数据体系，覆盖制造运营全链路核心场景，同时修复安灯列表页现有的 UI 渲染错位 Bug。**

### 1.2 用户故事

| 角色 | 用户故事 | 演示场景锚点 |
|------|---------|-------------|
| **操作工（张师傅）** | 作为产线操作工，我希望在发生异常时能通过安灯系统快速发起呼叫（缺料/设备故障/工艺异常），并能实时看到我的呼叫被谁响应、处理进展如何，以便我不用离开工位去"找人"。 | 安灯列表页缺料类 pending 记录 → 被班组长 acknowledged → 响应中 → 物料送到后 resolved |
| **班组长（王班）** | 作为班组长，我希望打开安灯看板能看见当前所有未解决的呼叫及其紧急程度、已超时时间，并能一键认领/转派/升级，确保我的班组不发生漏接。 | 安灯列表 Tab 切换（全部→待响应→已升级）→ 认领 → 升级 → 从列表看到流转 |
| **车间主任（李主任）** | 作为车间主任，我希望在驾驶舱和安灯页面查看本周/本月安灯趋势统计、解决率、平均响应时间、TOP 问题类型，以支撑我制定改善计划。 | KPI 卡片（待响应/响应中/今日已解决）有真实数据 → 统计面板有数字 |
| **运维工程师（陈工）** | 作为设备运维人员，我希望能看到设备台账全貌、运行状态分布、历史维修记录和保养计划，以便合理安排预防性维护和备件采购。 | 设备列表 30+ 台 → 按状态筛选（运行/空闲/维修/故障）→ 设备详情含维修历史 |
| **品质检验员（赵质检）** | 作为品质人员，我希望安灯系统中与质量相关的呼叫能关联到具体工单/产品批次，并能追踪从"质量待判定"到"已解决"的完整闭环，支持后续质量回溯。 | 安灯列表"质量待判定"类型 → 关联工单 → resolved 含 resolution 记录 |

### 2.1 设备 (equipment) — 目标 30+ 台

#### 车间 × 设备类型矩阵

| 车间/区域 | 设备类型 | 台数 | 品牌 | 状态分布 |
|-----------|---------|:----:|------|---------|
| **机加一区** | 立式加工中心 × 3、数控车床 × 2、卧式镗铣床 × 1 | 6 | FANUC、马扎克、沈阳机床 | running 3、idle 1、maintenance 1、fault 1 |
| **机加二区** | 五轴加工中心 × 2、数控车床 × 2、龙门铣 × 1 | 5 | DMG MORI、牧野、大连机床 | running 3、idle 1、fault 1 |
| **机加三区** | 精密磨床 × 2、数控铣床 × 2、电火花线切割 × 1 | 5 | 海天、沈阳机床 | running 2、idle 2、scrapped 1 |
| **焊接车间** | 焊接机器人 × 3、激光焊接机 × 1、氩弧焊工位 × 1 | 5 | ABB、KUKA、新松 | running 3、idle 1、maintenance 1 |
| **装配车间** | 装配工位 × 3、自动拧紧机 × 1、压力测试台 × 1 | 5 | — | running 2、idle 2、fault 1 |
| **注塑车间** | 注塑机 × 2、烘干机 × 1 | 3 | 海天 | running 1、idle 1、maintenance 1 |
| **质检中心** | 三坐标测量机 × 2、粗糙度仪 × 1、硬度计 × 1 | 4 | 蔡司、海克斯康 | running 2、idle 1、maintenance 1 |
| **物流区** | AGV × 3、自动立库 × 1 | 4 | 新松 | running 2、idle 1、maintenance 1 |

**状态分布汇总**：

| 状态 | 台数 | 占比 |
|------|:----:|:----:|
| running（运行中） | 18 | 46% |
| idle（空闲） | 9 | 23% |
| maintenance（保养/维修） | 5 | 13% |
| fault（故障） | 4 | 10% |
| scrapped（报废） | 1 | 3% |
| 预留 | 2 | 5% |
| **合计** | **39** | **100%** |

> 预留 2 台不作数据填充，用于演示"添加设备"功能。

**品牌覆盖**：FANUC、马扎克 (MAZAK)、DMG MORI、牧野 (Makino)、沈阳机床、大连机床、海天精工、蔡司 (ZEISS)、海克斯康 (Hexagon)、ABB、KUKA、新松 (SIASUN)

**关键字段要求**：每台设备必须填写 equipment_code（如 MC-01001）、name、category_id、model、manufacturer、install_date（2020-2025 区间）、location（含车间/行/列）、status、power_kw、parameters（JSON，含主轴转速/行程/精度等）。

## 3. 现有 Bug 描述

### 3.1 安灯列表页行渲染错位

#### 基本信息

| 项 | 值 |
|----|-----|
| **Bug ID** | BUG-ANDON-LIST-001 |
| **严重度** | 🔴 严重 |
| **优先级** | P0（影响演示效果，需立即修复） |
| **发现日期** | 2026-07-10 |
| **发现人** | 用户 + 产品检查 |

#### 现象描述

安灯列表页（路由 `/andon`）每行的 "呼叫#N"（左列）和 status 标签（右列）**垂直方向不对齐**。具体表现为：

- 用户截图可见：第 4 行的 "呼叫#4" 右列显示 `acknowledged`，但视觉上与第 3 行的 status 标签在同一水平线上，形成了"左列错位、右列乱序"的观感
- 每行的 title（呼叫#N）和 value（status）不在同一纵向基准线上
- 在 van-list 滚动加载更多数据时，错位现象在部分行加剧

#### 当前代码定位

`frontend/src/pages/andon/AndonList.vue:117-123`

```html
<van-cell
  v-for="item in calls" :key="item.id"
  :title="item.call_title || '呼叫#'+item.id"
  :label="item.source_desc || item.call_type"
  :value="item.status"
  is-link
  @click="goDetail(item.id)"
/>
```

#### 根因分析（假设，需工程师确认）

1. **Vant van-cell 组件的布局机制**：`van-cell` 内部使用 flex 布局，title 在左上方、label 在 title 下方、value 在右侧**垂直居中**。当各行的 label 行数不同（有的行 label 文本短不换行、有的行 label 文本长导致换行），value 的垂直居中基准会随行高变化，造成"不同行之间 title 与 value 不同基线"的视觉错位。

2. **另类可能原因**：CSS Grid 列数未固定，或 van-cell 在滚动容器中复用渲染样式时，部分行的 `.van-cell__value` 计算高度不同。

3. **排除因素**：不是 API 数据问题（即使数据正确也会渲染错位），不是后端响应问题。

#### 复现步骤

1. 访问 https://mfg1.ziwi.cn/andon
2. 确保列表存在至少 3 条以上安灯记录（当前种子数据即可）
3. 观察列表各行，聚焦"呼叫#N"文字左侧和 status 文字右侧的垂直位置
4. 可切换 Tab 到"全部""待响应""响应中"等分别确认

#### 验收标准

| # | 验收项 | 通过条件 |
|:-:|-------|---------|
| AC1 | 全部 Tab | 列表每行的"呼叫#N"、call_type、status 三者必须同行左右对应，视觉上同一水平线 |
| AC2 | 各个状态 Tab | 在"待响应/响应中/已解决/已升级"各 Tab 下渲染同样正确 |
| AC3 | 不同数据长度 | 当 title/label 文本长度不同时，列对齐仍然稳定 |
| AC4 | 滚动加载后 | 滚动触发更多数据加载后，新加入的行也不出现错位 |

#### 修复建议方向（工程师自行判断）

- **方案 A（推荐）**：将 van-cell 替换为自定 HTML 结构，使用 CSS Grid 或 flex 显式控制三列（呼叫信息 / call_type / status），使用 `align-items: flex-start` 确保列基线一致
- **方案 B**：保留 van-cell，通过 CSS 覆盖 `.van-cell__value` 的垂直对齐方式（`align-self: flex-start`）让 value 与 title 顶部对齐
- **方案 C**：使用 `<div>` + `v-for` 替代 van-cell，保持与 Vant van-list 的 scroll 事件兼容

#### 禁止事项（范围约束）

- ❌ 不做前端 API 改造
- ❌ 不更换组件库（van-cell 可覆盖样式，不替换为其他组件库）
- ❌ 不修改其他页面 UI
- ❌ 不改动后端接口
