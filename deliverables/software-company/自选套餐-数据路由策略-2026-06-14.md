## 一、核心原则

每个模块都应**可独立运行**，但当与其他模块配合时自动切换数据源。这个设计已经在产品方案中确立（§4.3 "轻量独立模式"），现在需要在技术层面落地。

```
模块 A 需要数据 X
  ├── 模块 B (数据 X 的提供者) 已启用 → 从模块 B 的 API/表 读取
  └── 模块 B 未启用               → 通过 M12 Excel 导入 + 自管理
```

## 三、代码实现策略

### 3.1 数据路由层（核心设计模式）

在每个会消费外部数据的 Repo 中，使用条件查询判断数据来源：

```python
# energy_repo.py — 设备数据路由示例
class EnergyRepository(MultiTenantRepository):

    async def list_eligible_devices(self, feature_flags: dict):
        """获取可用于能耗计算的设备列表（自动路由数据源）"""
        has_tpm = feature_flags.get("M02_EQUIPMENT", False)

        if has_tpm:
            # 场景：M02 已启用 → 从 equipment 表 JOIN 扩展信息
            return await self.query("""
                SELECT e.*, ed.device_type, ed.energy_type,
                       ed.rated_power, ed.multiplier
                FROM equipment e
                LEFT JOIN energy_device ed ON ed.equipment_id = e.id
                WHERE e.tenant_id = :_tenant_id
            """)
        else:
            # 场景：仅 M11 → 从 energy_device 自管表读取
            return await self.query("""
                SELECT ed.*, NULL as equipment_code, NULL as equipment_name
                FROM energy_device ed
                WHERE ed.tenant_id = :_tenant_id
            """)
```

### 3.2 告警路由（联动判断）

```python
# energy_repo.py — 告警联动安灯
class EnergyRepository(MultiTenantRepository):

    async def create_alert_with_routing(self, data: dict, feature_flags: dict):
        """创建能耗告警，根据 feature_flags 决定是否联动安灯"""
        alert_id = await self.create_alert(data)
        
        has_andon = feature_flags.get("M05_ANDON_CALL", False)
        should_link = has_andon and data.get("severity") in ("warning", "critical")
        
        if should_link:
            # 同进程调用 AndonRepository — 直接跨模块
            andon_repo = AndonRepository(self._session)
            andon_repo.set_tenant_id(self._tenant_id)
            await andon_repo.create_call({
                "tenant_id": self._tenant_id,
                "call_type": "equipment",
                "source": "auto",
                "caller_id": data.get("triggered_by", 0),
                "description": f"[能碳告警] {data.get('alert_name', '')} - {data.get('alert_message', '')}",
                "priority": "high" if data.get("severity") == "critical" else "normal",
            })
        
        return alert_id
```

### 3.3 生产数据接入（条件读取）

```python
# energy_repo.py — 碳核算时获取产量数据
class EnergyRepository(MultiTenantRepository):

    async def get_production_data_for_carbon(self, start_date, end_date, feature_flags: dict):
        """获取周期内产量数据用于计算碳排放强度"""
        has_mes = feature_flags.get("M01_WORK_REPORT", False)
        
        if has_mes:
            # 直接从 M01 的 work_report 表读取
            return await self.query("""
                SELECT SUM(output_qty) as total_output,
                       SUM(labor_hours) as total_hours
                FROM work_report
                WHERE report_date BETWEEN :start AND :end
            """, {"start": start_date, "end": end_date})
        else:
            # 无 M01，从碳核算扩展表或返回空
            return [{"total_output": 0, "total_hours": 0}]
```

### 3.4 FeatureFlag 注入方式

Feature flags 有两种注入方式：
1. **应用启动时加载** — 从 `feature_flags.json` 读取当前租户的套餐配置
2. **API 请求时传递** — 从 `current_user` 的套餐配置中提取

推荐方式：在 `dependencies.py` 的 `get_current_user` 或 `get_tenant_repo` 中注入 feature flags：

```python
# dependencies.py 扩展
async def get_feature_flags(current_user: dict = Depends(get_current_user)):
    """获取当前租户的模块启用状态"""
    tenant_id = current_user.get("tenant_id")
    # 从 DB 或 Redis 读取租户的套餐配置
    # ... (实现方式取决于套餐配置的存储位置)
    return {
        "M01_WORK_ORDER": True,
        "M02_EQUIPMENT": False,
        "M05_ANDON_CALL": True,
        "M11_ENERGY": True,
        # ...
    }
```

## 四、实现优先级

| 优先级 | 功能 | 说明 |
|:------:|:-----|:------|
| P0 | FeatureFlag 依赖注入 | 在 `dependencies.py` 中实现租户级 feature flag 注入 |
| P0 | M11 设备数据路由 | 能碳设备列表根据 M02 是否启用切换 JOIN 路径 |
| P0 | M11 告警→安灯联动 | 根据 M05 是否启用决定是否创建 andon_call |
| P1 | M01 产量数据路由 | 碳核算时根据 M01 是否启用读取产量数据 |
| P1 | M12 Excel 导入标准化 | 所有模块的"轻量模式"Excel 导入模板统一由 M12 提供 |
| P2 | TPM 设备同步脚本 | M02 启用时，自动创建 energy_device 扩展记录 |
| P2 | 供应商全平台化 | 供应商数据从 M11 专属提升为全平台共享模块 |
