## 三、具体改造方案

### 3.1 单点登录 (SSO) 整合

**目标**：能碳系统不再维护独立 JWT，复用 SaaS 平台的认证体系。

**现状问题**：
- 源系统 `auth.py` 有自己的 JWT secret、token 过期策略、密码哈希
- 源系统有独立的 `users` 表(存放于 SQLite)
- SaaS 平台已有完整的 JWT 认证+租户隔离

**改造方案**：

| 层级 | 改造内容 | 工作量 |
|:-----|:---------|:------:|
| 后端 | 能碳 API 路由全部改用 SaaS 的 `get_current_user` + `get_tenant_repo` | ✅ 已完成 |
| 后端 | 移除源系统 `auth.py` 和 `auth_router.py` 的独立认证逻辑 | 🔲 待清理 |
| 后端 | 添加跨域/CORS 配置支持能碳前端独立域名访问 SaaS API | 🔲 需配置 |
| 前端 | 能碳前端登录页指向 SaaS 统一登录 `/login` | 🔲 待改造 |
| 前端 | 能碳前端使用 SaaS 返回的 `access_token` 调用 API | 🔲 待改造 |
| 前端 | 存量的能碳用户需迁移或映射到 SaaS 用户表 | 🔲 需数据迁移 |

**SSO 流程**：
```
用户 → SaaS登录页 → 获取JWT Token
       ↓
   能碳前端 → 用 Token 调用 /api/v1/energy/* 
       ↓
   SaaS 后端验证 Token → 提取 tenant_id → 返回能碳数据
```

---

### 3.2 设备数据同步 (知微云 IoT)

**目标**：能源设备不再各自创建，从知微云 IoT 平台获取设备列表。

**现状问题**：
- 源系统的 `devices` 表(工作中心+设备)是独立维护的
- 设备 IoT 数据通过模拟器 `energy_simulator.py` 生成，非真实数据
- SaaS 平台的 `equipment` 表(TPM)已有生产设备台账

**改造方案**：

| 子项 | 方案 | 优先级 |
|:-----|:-----|:------:|
| **设备映射** | `energy_device.equipment_id` → 关联 `equipment.id`(TPM设备)，能源设备作为生产设备的"计量附属" | P0 |
| **同步机制** | 创建 `EnergySyncService`，定时/手动从 `equipment` 表同步设备到 `energy_device` 表 | P0 |
| **IoT 数据源** | 创建 IoT 数据接入层，支持 MQTT/API 拉取两种模式 | P1 |
| **时序数据** | `energy_meter_data` 表(已建)使用 TimescaleDB 超表存储 IoT 时序数据 | P1 |
| **数据回写** | 能碳系统计算的碳排放数据可回写到生产模块的报表 | P2 |

**设备同步流程**：
```
TPM 设备台账 (equipment表)
       │
       ▼
  EnergySyncService.sync_devices()
       │
       ▼
  energy_device 表
  (自动建立 equipment_id 关联)
       │
       ▼
  IoT 网关采集数据 → energy_meter_data (时序)
       │
       ▼
  碳排放核算 → carbon_emission_record
```

### 3.3 供应商与生产数据同步

**目标**：供应商数据和能耗数据从主平台获取，避免双写。

**现状问题**：
- 源系统有独立的 `suppliers` 表和管理页面
- 供应商碳数据(月度报表)要求供应商自行上传
- SaaS 平台的 `organization` 模块已有员工/团队管理

**改造方案**：

| 子项 | 方案 | 优先级 |
|:-----|:-----|:------:|
| **供应商数据** | 若 SaaS 平台无独立供应商模块 → 保留能碳系统供应商管理，但 `tenant_id` 隔离 | P0 |
| **生产数据关联** | `carbon_emission_record.factory_id` / `energy_device.factory_id` 关联到组织架构 | P1 |
| **月度数据采集** | 保留供应商月度碳数据导入功能(Excel)，数据源归属到 tenant_id | P1 |
| **数据同步触发** | 在 `production` 模块的报工/工单完成后，可触发能碳计算 | P2 |

### 3.4 能碳预警 → 安灯联动

**目标**：能耗/碳排放告警触发时，自动创建安灯呼叫。

**现状问题**：
- 源系统的 `monitoring.py` 有告警逻辑但未与安灯系统关联
- SaaS 平台的 `energy_alert` 表和 `andon_call` 表各自独立
- 目前已在 `api/energy.py` 创建了告警 CRUD，但无联动逻辑

**改造方案**：

```
energy_alert 触发 (severity=critical/warning)
       │
       ▼
  EnergyAlertService.check_and_link_andon()
       │
       ▼
  创建 andon_call
  call_type = 'equipment' 
  source = 'auto' (自动触发)
  description = 告警描述
  priority = 告警 severity 映射
```

| 子项 | 方案 | 优先级 |
|:-----|:-----|:------:|
| **告警→安灯映射** | severity=critical → 自动创建andon_call (priority=high/emergency) | P0 |
| **联动Service** | 创建 `EnergyAlertService`，在告警创建后调用 `AndonRepository.create_call()` | P0 |
| **状态同步** | 安灯解决后 → 回写能耗告警状态为 resolved | P1 |
| **超时升级** | 安灯响应超时规则可关联到能耗告警升级 | P2 |
