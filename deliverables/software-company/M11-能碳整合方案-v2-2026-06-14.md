## 1. 核心认知修正声明

### 1.1 原 v2 文档的前提错误

| 项目 | v2 文档的错误认知 | 客户（Kane）的实际需求 |
|:-----|:-----------------|:---------------------|
| **前提** | "能碳是知微云的一个模块，不是两个系统之间的数据同步，而是同一个系统的模块间数据引用" | "我的想法还是**两个独立系统相互同步**。能碳可能独立上，所以需要独立的轻量化输入输出机制" |
| **部署模型** | 所有模块同库/同部署/同认证 | M11 能碳是**独立可部署的系统**，可单独运行 |
| **数据关系** | `LEFT JOIN` 条件路由，同一 DB 内切换 | 两个系统之间做同步，各自有**独立 DB** |
| **轻量化 I/O** | 未考虑 Excel 导入兜底 | Excel 导入是**独立模式的核心能力**（日更电表读数） |
| **独立看板** | 未涉及 | 能碳独立看板是**独立模式的标配** |

> 引用：评审输入 §一、偏差定位

### 1.2 核心不等式（架构师评审 §附录）

```
修正前（错误认知）:
  M11 = 知微云平台的一个模块
  → 同库 JOIN / 条件路由 / FeatureFlag / 函数调用
  → DataRouteResolver 万事通

修正后（正确认知）:
  M11 ≠ 知微云平台的一个模块
  M11 = 能碳独立系统 ↔⟨Event Push + API Pull⟩↔ 知微云平台

  同平台模块（B/C/D 类）:
    → 同库 JOIN / 条件路由 / FeatureFlag / 函数调用 ✅
    → DataRouteResolver 在这里用 ✅

  跨系统同步（A 类 M11 ↔ 平台）:
    → API 同步 / 事件同步 / Excel 导入 ❌ 不同库
    → DataRouteResolver 在这里无用 ❌
```

> 引用：架构师评审 §附录：核心不等式

### 1.3 修正方向

| 维度 | v2 方案 | v3 修正方案 |
|:-----|:--------|:-----------|
| M11 定位 | 知微云的一个模块 | **独立系统**，通过同步层与知微云对接 |
| 数据架构 | 同库 JOIN（equipment LEFT JOIN energy_device） | 独立 DB（SQLite/PG）+ Hybrid Sync 同步层 |
| 数据路由 | DataRouteResolver 用于所有模块 | DataRouteResolver 仅用于同平台模块（B/C/D 类） |
| 数据入口 | 平台 API 调用 | 独立模式：Excel 导入；平台模式：Event Sync |
| 独立看板 | 未规划 | M11 同部署同数据源的内嵌看板 |
| 认证 | 全平台统一认证 | 独立模式：独立认证；平台模式：SSO |

## 2. 三层模块分类模型

> 基于 PM 评审建议（`review-pm-correction-2026-06-14.md` §一）修正后的四类模型

### 2.1 类型 A：真独立模块

| 模块 | 分类依据 | 数据架构 |
|:-----|:---------|:---------|
| **M11 能碳管理** | 可单独卖给客户，自成业务闭环 | 独立 DB（SQLite/PG）、自有认证、Excel I/O |

**特征**：
- 可以单独部署，不依赖知微云平台的其他模块
- 自有数据库：**SQLite**（轻量独立模式）或 **PostgreSQL**（高可用模式）
- 自有认证：`standalone_auth`（独立 JWT，不依赖平台 SSO）
- 核心数据入口：**Excel 导入**（日更电表读数等）
- 自有看板：能碳 Dashboard / 大屏（同部署同数据源）
- 有平台支撑时：数据从平台同步而来，Excel 导入降级为兜底

**独立部署的真实场景验证**（PM 评审 §三）：

| 客户画像 | 典型场景 | 驱动因素 |
|:---------|:---------|:---------|
| 碳合规驱动型 | 出口欧盟企业，需提交碳排放报告 | CBAM（碳边境调节机制）、客户供应链碳审计 |
| 集团能碳管控型 | 集团总部要管各工厂能耗碳排 | 集团 KPI、双碳目标、ESG 报告 |
| 政府监管驱动型 | 重点用能单位需按时上报能耗数据 | 《重点用能单位节能管理办法》 |
| 已有 MES/ERP 的客户 | 已有生产管理系统，只缺能碳 | 用友/金蝶/Odoo 客户，只买能碳模块 |
| 物业/园区管理 | 非制造业但也需要能碳管理 | 园区能耗管理、碳排放核算 |

> 引用：PM 评审 §三、业务场景验证——**"完全成立"**

### 2.2 类型 B：强建议依赖

| 模块 | 建议依赖 | 降级方案 |
|:-----|:---------|:---------|
| **M05 安灯 → M02 TPM** | 🟡 **建议**（非必须） | 缺料/质量/工艺安灯**不依赖 TPM**，纯安灯场景（拉绳报警）完全独立 |
| **M03 品质 → M01 生产** | 🟡 **建议**（非必须） | IQC 来料检验独立于生产，出货检验独立运行 |

**重要修正**：原评审输入的"🔴 必须"标注已被 PM 评审降级为"🟡 建议"。

**制造业实战验证**（PM 评审 §一、偏差1）：
- **纯安灯场景**：电子组装厂只需"拉绳报警"系统，工人按铃→班长回应→记录，**根本不需要设备管理**
- **纯品质场景**：IQC 来料检验是独立于生产的，小型工厂可能先上品质模块管 IQC，生产模块后面再上

> 引用：PM 评审 §一、偏差1——"建议降级为'建议依赖'"

### 2.3 类型 C：半独立模块

| 模块 | 独立模式描述 | 降级方案 |
|:-----|:------------|:---------|
| **M01 生产管理** | 手工排产 + 个人报工 | 无 IoT 时手动录入产量 |
| **M06 ERP** | 轻量进销存独立运行 | 无 WMS/CRM/SRM 时独立运行 |
| **M07 WMS** | 出入库单据手动创建 | 无 ERP 时手工录入单据 |
| **M10 轻量财务** | 财务数据 Excel 导入 | 无进销存数据时手工填表 |
| **M13 看板** | 空白看板占位 | 无数据源时创建空看板 |

**统一轻量独立模式规范**（PM 评审建议 2）：

```yaml
轻量独立模式规范：
  数据入口: Excel 导入 或 手动录入
  数据格式: 与标准模式完全一致（导入即兼容）
  升级路径: 开启自动联动开关即可（无需数据迁移）
  界面表现: 显示"轻量模式"标签，提示"升级到标准模式可自动对接XX模块"
```

> 引用：PM 评审 §一、偏差2——"类型 C 遗漏了多个'半独立'模块"

|:-----|:-----|:---------|

| **M13 看板** | 需要数据源但允许空白占位 | 客户先买看板做布局，后续加数据源 |

> 引用：PM 评审 §一、偏差3——"新增类型 D：基础设施/横向服务"

### 2.5 类型演变矩阵（M11 的演进路径）

| 阶段 | M11 的"类型" | 数据架构 | 部署方式 |
|:-----|:------------|:---------|:---------|
| **Phase 1（Now）** | 类型 A（真独立） | 独立 DB，API 同步 | 独立部署或与平台同部署 |
| **Phase 2（Next）** | 类型 A→C 过渡（半独立） | 共享 M12 数据层，独立认证 | 仍可独立，但推荐集成 |

### 3.2 Phase 1 轻量方案：本地 change_log 表 + 定时拉取

**无需消息中间件**，0 额外基础设施依赖。

```
知微云平台                          M11 能碳系统
──────────                          ────────────
数据变更
    │
    ▼
写入本地 change_log 表              HTTP POST /api/v1/sync/pull
(记录: table, row_id, action, ts)  (M11 定时轮询, 默认 60s)
    │                                    │
    └─── 等待 M11 拉取 ──────────────────▶
                                         │
                                         ▼
                                    M11 解析 change_log
                                    → 调用平台 API 获取完整数据
                                    → 写入本地 DB

定时对账(每日 02:00):
  平台: SELECT count(*) FROM equipment WHERE updated_at > last_sync
  M11: 对比本地数据量 → 不一致时全量拉取
```

**change_log 表结构**：

```sql
CREATE TABLE sync_change_log (
    id          BIGSERIAL PRIMARY KEY,
    table_name  VARCHAR(64) NOT NULL,      -- 变更表名 (equipment, supplier, work_report)
    row_id      BIGINT NOT NULL,           -- 变更行 ID
    action      VARCHAR(16) NOT NULL,       -- INSERT / UPDATE / DELETE
    changed_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    processed   BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_change_log_processed ON sync_change_log(processed, changed_at);
```

> 引用：架构师评审 §1.2——Phase 1 轻量实现方案

### 3.3 Phase 2 增强方案：消息队列 + T+1 对账

| 组件 | 技术选型 | 用途 |
|:-----|:---------|:-----|
| 消息队列 | Redis Stream / RabbitMQ | 可靠事件推送，天然背压和重试机制 |
| 对账服务 | `sync_reconciler.py` | 每日 02:00 全量对账 + 不一致修复 |
| 监控告警 | 对账失败 → 告警通知 | 数据一致性的兜底保障 |

### 3.4 同步方向与数据主权矩阵

基于**"谁创建谁主责"**原则：

| 数据类型 | 数据主权方 | M11 角色 | 同步方向 |
|:---------|:----------|:---------|:---------|
| **设备档案** | **M02 TPM**（平台设备台账是唯一权威来源） | 🔴 **只读消费** | M02 → M11（单向） |
| **供应商主数据** | **M01 生产管理** | 🔴 **只读消费** | M01 → M11（单向） |
| **产量数据（报工）** | **M01 生产管理** | 🔴 **只读消费** | M01 → M11（单向） |
| **能耗数据** | **M11 能碳** | 🟢 **数据所有者** | M11 自有，不同步回平台 |
| **碳排放核算结果** | **M11 能碳** | 🟢 **数据所有者** | M11 自有 |
| **能耗告警** | **M11 能碳** | 🟢 **数据所有者** | M11 自有（告警可联动触发平台安灯） |
| **IoT 时序数据** | **M12 IoT 网关** | 🟡 **半消费** | M12 → M11（若 M12 存在且连接） |

### 3.5 冲突策略

| 场景 | 冲突类型 | 解决策略 |
|:-----|:---------|:---------|
| 平台修改了设备名称，M11 也修改了同一设备名称 | 双向修改冲突 | **平台覆盖 M11**（设备主权在平台） |
| 平台删除了设备，M11 有该设备的能耗记录 | 引用完整性冲突 | **软删除**：平台标记删除 → 同步到 M11 标记为"已停用"（保留历史能耗数据） |
| M11 通过 Excel 导入的设备，平台没有 | 数据孤岛 | **不自动反向同步**。Excel 导入设备仅存在于 M11 |
| 平台和 M11 同时新增了设备 | 重复创建 | **以平台编码为准**，通过 `sync_id` 和 `source` 字段区分来源 |

### 3.6 同步频率

| 数据类型 | 主通道 | 频率 | 兜底对账 |
|:---------|:-------|:-----|:---------|
| **设备变更** | Event Push（实时） | 秒级 | T+1 全量对账 |
| **供应商变更** | Event Push（实时） | 秒级 | T+1 全量对账 |
| **产量报工** | Event Push（准实时） | 分钟级 | T+1 全量对账 |
| **能耗数据** | 无（M11 自有） | — | — |
| **IoT 数据** | Event Push（若 M12 存在） | 秒级 | T+1 全量对账 |

### 4.2 独立 DB

- **轻量模式**：SQLite（WAL 模式，支持读写并发）
- **生产模式**：PostgreSQL（高可用）
- **数据来源跟踪**：所有表增加来源字段

```sql
-- M11 本地 energy_device 表补充字段
ALTER TABLE energy_device ADD COLUMN data_source VARCHAR(20) DEFAULT 'excel';
  -- 'excel' = Excel 导入（独立模式）
  -- 'sync'  = 从平台同步（全平台模式）
  -- 'manual' = 手动录入

ALTER TABLE energy_device ADD COLUMN source_id VARCHAR(64);
  -- 平台端对应记录的 ID（如 equipment.id），用于同步关联

ALTER TABLE energy_device ADD COLUMN sync_version INTEGER DEFAULT 0;
  -- 乐观锁，用于检测同步冲突
```

> 引用：架构师评审 §3.4——M11 本地 DB 的数据模型补充

### 4.3 独立认证（standalone_auth）

- 独立 JWT 签发（不依赖平台 SSO）
- 本地用户表（admin / operator / viewer 三角色）
- 可选后续对接 SSO（Phase 2）
- 双模式切换：`standalone=True` 时使用独立认证，`standalone=False` 时使用平台 SSO

```python
# POST /api/v1/auth/login  (独立模式)
{
    "username": "admin",
    "password": "***"
}
# 返回:
{
    "access_token": "<jwt_token>",
    "token_type": "bearer",
    "mode": "standalone"
}

# 平台模式：走平台 SSO 登录流程（/api/v1/auth/sso/login）
```

### 4.4 Excel 导入为核心数据入口

- **日更操作**：电表读数每日导入（独立模式核心）
- **周更操作**：水表/气表/蒸汽表
- **月更操作**：月度能源账单
- **一次性/批量**：设备档案导入、供应商导入、碳排放因子导入

具体见 §5 轻量化 I/O 机制。

### 4.5 独立看板（同部署同数据源）

| 方案 | 描述 | 评价 |
|:-----|:-----|:-----|
| **A. 内嵌看板（推荐）** | 看板组件与 M11 系统同部署，直接从 M11 本地 DB 查询 | ✅ 最简单，离线可用，数据权限清晰 |

### 5.3 三层校验规则

```
Layer 1: 模板格式校验 (File Level)
  ├── 文件是否是有效的 .xlsx/.xls
  ├── 模板编码是否匹配
  ├── 表头列名是否与模板定义一致
  └── 必填列是否缺失
        ↓ 通过
Layer 2: 字段级校验 (Field Level)
  ├── 数据类型校验（数值/字符串/日期）
  ├── 格式校验（正则表达式）
  ├── 枚举值校验（能源类型是否在允许列表）
  ├── 长度校验
  └── 唯一性校验（如设备编码在本次导入中不重复）
        ↓ 通过
Layer 3: 业务逻辑校验 (Business Level)
  ├── 设备编码是否与现有数据冲突（DB 查重）
  ├── 关联数据存在性校验（如引用的设备 ID 是否存在）
  ├── 阈值校验（如能耗值是否在合理范围内）
  └── 业务规则校验（如结束时间 >= 开始时间）
        ↓ 通过
写入数据库
```

> 引用：架构师评审 §4.3——三层校验规则

### 5.4 PM 评审的 UX 建议

能碳独立模式下，日更 Excel 导入是高频操作，UI 应支持（PM 评审 §3.2）：

1. **最近使用的模板**快速入口
2. **导入进度和校验结果**的实时反馈（行数校验、阈值校验）
3. **上次导入数据与本次数据的差异对比**（如"今日用电量较昨日+15%"，提示可能异常）

### 7.2 `route_resolver.py` — 保留，范围缩小

**评估结论**：设计质量良好（声明式路由优于 if/else），但适用范围需明确缩小。

| 适用性 | 判定 |
|:-------|:-----|
| ❌ 不用于 M11 跨系统同步 | 独立系统间不存在同库 JOIN |
| ✅ 用于同平台模块（B 类：M05→M02 安灯联动路由） | 同库条件路由 |
| ✅ 用于 C 类（M01 轻量模式降级路由） | 同库条件路由 |
| ✅ 用于 D 类（M12/M13 数据来源路由） | 同库条件路由 |

**需修改**：在类注释中明确：
```python
"""
此路由解析器仅用于同平台模块（B/C/D 类），不适用于跨独立系统同步。
M11 能碳独立模式不使用此路由解析器。
"""
```

> 引用：架构师评审 §2.2——`route_resolver.py` 评估

### 7.3 `energy_repo.py` — 重构

**核心问题**：当前代码走同库 JOIN 路径（`equipment LEFT JOIN energy_device`），但独立系统架构下应统一从 M11 本地 DB 读。

```
修正前 (同库 JOIN):
  list_devices():
    if has TPM:  SELECT FROM equipment JOIN energy_device  (同库)
    else:        SELECT FROM energy_device                   (同库)

重构后 (统一从 M11 本地 DB 读):
  list_devices():
    # 无论数据来源是 Excel 导入还是事件同步
    # 数据都已经在 M11 本地 DB 的 energy_device 表中
    SELECT FROM energy_device WHERE ...
    # 不需要管数据是怎么来的（由同步层负责）
```

**写入路径**才是需要根据模式切换的：

```python
create_device():
  if STANDALONE_MODE:
      # 写入 M11 本地 energy_device 表
      INSERT INTO energy_device ...
  else:
      # 平台模式：写入平台 equipment 表（让平台处理）
      # 通过平台 API 创建设备 → 平台 Event Push 回来 → M11 消费
      await self._platform_api.create_equipment(data)
```

| R6 | **SQLite 并发写入冲突** | 🟡 中 | SQLite WAL 模式；导入操作放在低峰期；后续可升级为 PostgreSQL |
| R7 | **模板版本不兼容** | 🟡 中 | 模板版本管理（SemVer）；向后兼容策略 |
| R8 | **T+1 对账发现不一致但无法自动修复** | 🟢 低 | 自动修复规则（以平台数据为准）；无法修复的标记"异常"并告警 |

> 引用：架构师评审 §六——风险矩阵
